# Phase 09.6: UX Safety Gates - Research

**Researched:** 2026-03-12
**Domain:** Vue 3 overlay UI patterns + Rust atomic hotkey swap
**Confidence:** HIGH

## Summary

Phase 09.6 is a two-part UX hardening phase with no external library dependencies — both changes are pure modifications to existing code files. The confirmation gate for shutdown/restart is a Vue overlay following the exact same structural pattern already used by the context menu (`v-if`, `position: fixed`, `@mousedown.prevent`). The hotkey atomic swap is a four-line reorder in `hotkey.rs::update_hotkey` plus removal of the runtime Alt+Space fallback path from that function (keeping startup fallback in `register()` intact).

The main planning risk is overlay focus management: the Tauri window's `onFocusChanged` auto-hides the launcher on any focus loss, and the confirmation overlay's mousedown interactions must use `@mousedown.prevent` to avoid triggering the focus-loss path. The context menu already solves this exact problem — the confirmation overlay inherits the same guard.

**Primary recommendation:** Model the confirmation overlay 1:1 on the existing context menu overlay structure (backdrop div + card div, `@mousedown.prevent` on interactive buttons, `v-if` driven by `confirmPending` ref). The hotkey swap is a straightforward reorder with removal of the fallback branch from `update_hotkey`.

## Architecture Patterns

### Relevant Existing Pattern: Context Menu Overlay

The existing context menu (App.vue lines 558-573) establishes the template to follow:

```html
<!-- Backdrop: click-outside closes overlay -->
<div v-if="menuVisible" class="menu-backdrop" @mousedown.prevent="closeMenu"></div>

<!-- Overlay card: positioned, interactive buttons use @mousedown.prevent -->
<div v-if="menuVisible" class="context-menu" :style="{ left: menuX + 'px', top: menuY + 'px' }">
  <div class="menu-item" @mousedown.prevent="openSettings">Settings</div>
  <div class="menu-item" @mousedown.prevent="quitApp">Quit Launcher</div>
</div>
```

Key guard: `@mousedown.prevent` is used (NOT `@click`) because the launcher's `onFocusChanged` auto-hide fires on mousedown. Using `@click` would allow focus loss to trigger before the click handler runs, causing the window to hide before the action completes.

### Confirmation Overlay Pattern

```
App.vue state additions:
  confirmPending: ref(false)
  pendingCommand: ref<SearchResult | null>(null)

Flow:
  Enter on system:shutdown or system:restart
    → set pendingCommand = item
    → set confirmPending = true
    → (do NOT call run_system_command yet)

  Overlay renders:
    - Full-coverage dim backdrop (position: fixed, inset: 0)
    - Centered card with title + body per pendingCommand.id
    - Confirm button (accent, focused via nextTick + confirmBtnRef.value?.focus())
    - Cancel button (subdued)

  ESC key while confirmPending:
    → confirmPending = false
    → pendingCommand = null
    → inputRef.value?.focus() to restore input focus

  Enter key while confirmPending:
    → call confirmAction()

  Confirm button @mousedown.prevent:
    → call confirmAction()
    → confirmAction: set confirmPending=false, invoke run_system_command, hideWindow()

  Cancel button @mousedown.prevent:
    → confirmPending = false
    → pendingCommand = null
    → inputRef.value?.focus()
```

### ESC Key Handler Integration Point

The existing `onKeyDown` Escape handler (App.vue line 255-263) already has a priority chain:
1. `if (menuVisible.value)` → close menu
2. else → hideWindow()

The confirmation overlay adds a new priority at position 1.5:

```typescript
if (confirmPending.value) {
  cancelConfirm()
  return
}
```

This must be inserted between the menu-close branch and the hideWindow() branch.

### Enter Key Handler Integration Point

The existing Enter handler (App.vue lines 277-288) calls `launchItem(item)` unconditionally. The confirmation intercept wraps the system command path:

```typescript
case 'Enter': {
  e.preventDefault()
  const item = results.value[selectedIndex.value]
  if (!item) break
  if (e.ctrlKey && e.shiftKey) {
    launchElevated(item)
  } else {
    launchItem(item)  // launchItem checks item.kind === 'system' + needs confirmation
  }
  break
}
```

The `launchItem` function currently invokes `run_system_command` directly for system items. The change: intercept shutdown/restart before the invoke, show the overlay instead.

### Two-Phase Hotkey Swap Pattern

Current `update_hotkey` in `hotkey.rs` (lines 56-80):
1. Unregister old hotkey
2. Call `register()` (which itself has fallback logic)
3. If registered != requested → return Err

Problem: if step 2 fails, the old hotkey is already gone.

New sequence:
1. Attempt to register new hotkey FIRST (using `on_shortcut` directly, not `register()`)
2. If registration fails → return `Err(...)`, old hotkey unchanged
3. If registration succeeds → unregister old hotkey (log failure, don't roll back)
4. Persist and return `Ok(())`

The runtime `update_hotkey` must NOT call `register()` (which contains the startup fallback logic). Instead it calls `app.global_shortcut().on_shortcut(...)` directly, mirroring the handler setup in `register()`, but without the recursive fallback.

```rust
// Pseudocode for new update_hotkey logic
pub fn update_hotkey(app, hotkey, data_dir) -> Result<(), String> {
    let settings = get_settings(&app, &data_dir);
    let old_hotkey = settings.hotkey.clone();

    // Step 1: Register new hotkey first
    let win = app.get_webview_window("launcher")
        .ok_or_else(|| "launcher window not found".to_string())?;
    let win_clone = win.clone();

    app.global_shortcut()
        .on_shortcut(&hotkey, move |_app, _shortcut, event| {
            if event.state == ShortcutState::Pressed {
                if win_clone.is_visible().unwrap_or(false) {
                    let _ = win_clone.hide();
                } else {
                    let _ = win_clone.show();
                    let _ = win_clone.set_focus();
                    let _ = win_clone.emit("launcher-show", ());
                }
            }
        })
        .map_err(|e| format!("'{}' could not be registered: {}", hotkey, e))?;

    // Step 2: Unregister old (non-fatal)
    app.global_shortcut()
        .unregister(&old_hotkey)
        .unwrap_or_else(|e| eprintln!("[hotkey] unregister old failed: {}", e));

    // Step 3: Persist
    let mut new_settings = settings;
    new_settings.hotkey = hotkey;
    set_settings(&app, &data_dir, &new_settings);

    Ok(())
}
```

### Settings.vue Inline Error Pattern

The `onHotkeyChange` function in Settings.vue currently reloads settings from backend on error. The new requirement adds an inline error message below the KeyCapture field.

Add a `hotkeyError` ref:
```typescript
const hotkeyError = ref<string | null>(null)
```

In `onHotkeyChange`:
- On success: clear `hotkeyError`
- On error: extract the error message, set `hotkeyError`, revert `settings.value.hotkey` to old value (the displayed KeyCapture will revert via its prop watcher)

In template, after `<KeyCapture>`:
```html
<p v-if="hotkeyError" class="hotkey-error">{{ hotkeyError }}</p>
```

KeyCapture already watches `props.modelValue` and reverts `displayValue` when the prop changes outside capture mode (line 11 of KeyCapture.vue). So setting `settings.value.hotkey` back to the old value will automatically revert the UI display.

### Anti-Patterns to Avoid

- **Using `@click` instead of `@mousedown.prevent` on overlay buttons:** Focus-loss fires on mousedown; `@click` arrives after the window would already be hiding. The context menu pattern (`@mousedown.prevent`) is mandatory.
- **Calling `register()` from `update_hotkey`:** `register()` contains the startup fallback to Alt+Space. The runtime swap must use `on_shortcut` directly so a failed new registration leaves the old hotkey active (not replaced with Alt+Space).
- **Hiding the launcher before confirmation:** `run_system_command` in Rust currently hides the launcher on entry. This must NOT be changed in Rust — the frontend must simply not invoke `run_system_command` for shutdown/restart until after confirmation. The Rust side change is to remove the upfront `win.hide()` for commands that require confirmation, OR (simpler) the frontend hides the window itself via `hideWindow()` before invoking (matching the existing `launchItem` pattern).
- **Losing query/selected result on cancel:** `confirmPending`/`pendingCommand` must NOT modify `query` or `results` or `selectedIndex`. Cancel must return state to exactly what it was before the overlay appeared.
- **`autoFocus` HTML attribute on confirm button:** Does not work reliably in Tauri webviews. Use `nextTick(() => confirmBtnRef.value?.focus())` after setting `confirmPending = true`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Overlay backdrop | Custom click-tracking logic | `v-if` div with `@mousedown.prevent` on backdrop | Already proven in context menu; avoids event ordering bugs |
| Focus trap in overlay | Custom tab/focus management | Focus confirm button via `nextTick` + ref; Escape handled in existing `onKeyDown` | Minimal overlay — only two buttons, no tab-trap complexity needed |
| Hotkey registration | Custom OS API calls | `tauri_plugin_global_shortcut` `on_shortcut` | Already integrated; handles cross-platform shortcut parsing |

## Common Pitfalls

### Pitfall 1: Focus Loss Race on Mousedown
**What goes wrong:** If overlay buttons use `@click`, the `onFocusChanged` listener fires on mousedown (before click), causing `hideWindow()` to run before the button handler. The overlay disappears without confirming.
**Why it happens:** Tauri's focus-change event fires on the OS mousedown event, which precedes the browser `click` event.
**How to avoid:** Use `@mousedown.prevent` on all interactive overlay elements. The `prevent` stops the browser from moving focus away from the launcher window, so `onFocusChanged` does not fire. This is exactly what the context menu backdrop and menu items do.
**Warning signs:** Overlay disappears immediately when clicking confirm/cancel buttons during testing.

### Pitfall 2: Double-Enter Triggering Confirmation Then Confirm
**What goes wrong:** User presses Enter on a search result, the overlay appears, and the same keydown event or a second rapid keydown triggers the confirm action — skipping the deliberate second press.
**Why it happens:** The overlay opens synchronously in response to Enter; if the button is focused before the keydown event finishes bubbling, Enter may fire on the button.
**How to avoid:** The `@mousedown.prevent` on the confirm button means keyboard Enter on the focused button fires `@click` (not `@mousedown`), so the separate keydown handler on the launcher handles Enter→confirm. The overlay's Enter path goes through `onKeyDown` (which checks `confirmPending`), not through a button click. The context says this is not a concern because overlay open + second deliberate Enter is the intended UX — but verify in manual testing.
**Warning signs:** Commands execute without user seeing the overlay.

### Pitfall 3: `register()` Called from `update_hotkey` Introduces Startup Fallback at Runtime
**What goes wrong:** The current `update_hotkey` calls `crate::hotkey::register(&app, &hotkey)` which has recursive fallback to Alt+Space. If the new hotkey fails, it silently registers Alt+Space instead of returning Err.
**Why it happens:** `register()` was designed for startup (where fallback is acceptable). Runtime updates need true rollback.
**How to avoid:** In the new `update_hotkey`, use `app.global_shortcut().on_shortcut(...)` directly — bypass `register()` entirely. Return `Err` if it fails, leave old hotkey registered.
**Warning signs:** User sets a bad hotkey → Settings shows an error → actual hotkey is Alt+Space, not the previous working key.

### Pitfall 4: System Command Rust Side Hides Launcher Before Frontend Confirmation
**What goes wrong:** The current `run_system_command` calls `win.hide()` immediately on entry (system_commands.rs line 8-10). If the frontend invokes it only after confirmation, this is fine. But this means any future code path that calls `run_system_command` directly for shutdown/restart will skip the confirmation gate.
**Why it happens:** The hide was added to ensure the launcher disappears before the destructive action.
**How to avoid:** The frontend's `launchItem` path (which is the only caller) will handle `hideWindow()` before invoking. The Rust side hide is a safety net — it is acceptable to leave it in place, since the confirmation overlay means the frontend will have already hidden the window by the time invoke reaches Rust. No Rust change is strictly required for correctness. Alternatively, remove the upfront hide from Rust for confirmation-required commands if cleaner separation is desired. The planner should decide.
**Warning signs:** Launcher briefly flashes hidden/shown if Rust hides before the animation completes.

### Pitfall 5: KeyCapture `displayValue` Not Reverting After Error
**What goes wrong:** After a hotkey error, `settings.value.hotkey` is reverted in the script but the `KeyCapture` component still shows the rejected key.
**Why it happens:** `KeyCapture` watches `props.modelValue` and only reverts `displayValue` when `!capturing.value` (line 11). If `capturing` is false at the time, the revert should work automatically.
**How to avoid:** Setting `settings.value.hotkey = oldHotkey` on error will update the `v-model` binding on `<KeyCapture>`, triggering the watcher which sets `displayValue = v` (since `capturing.value` is false after the capture completes). This is the correct path — just verify in testing.
**Warning signs:** KeyCapture still shows the rejected hotkey string after the error message appears.

## Open Questions

1. **Rust-side hide in run_system_command**
   - What we know: `system_commands.rs` line 8-10 hides the launcher immediately on entry for ALL system commands including lock/sleep (which don't go through confirmation). The frontend's `launchItem` currently calls `hideWindow()` AFTER `invoke('run_system_command', ...)` returns. Since shutdown/restart actually execute asynchronously via `Command::spawn()`, the `invoke` returns before the system shuts down. The frontend then calls `hideWindow()`. There's a timing overlap: Rust hides AND frontend hides.
   - What's unclear: Should the Rust-side upfront hide be removed for the two confirmation-gated commands? Or left as a safety net (double-hide is harmless)?
   - Recommendation: Leave the Rust-side hide in place for all commands. For confirmation-gated commands, the frontend will call `hideWindow()` in `confirmAction()` before invoking. The Rust hide will be a no-op (already hidden). This is the simplest approach with no Rust changes to system_commands.rs.

2. **Confirm button focus in Tauri webview**
   - What we know: `autoFocus` HTML attribute behavior in Tauri/Chromium webviews is reliable for form elements, but the pattern used elsewhere in this project (e.g., `inputRef.value?.focus()` in the `launcher-show` listener) uses explicit ref-based focus via `nextTick`.
   - What's unclear: Whether `autofocus` attribute on the confirm button would work in the confirmation overlay.
   - Recommendation: Use explicit `nextTick(() => confirmBtnRef.value?.focus())` after `confirmPending.value = true` — consistent with the established pattern in this codebase.
