# Phase 09.6 Plan 01: Atomic Hotkey Swap and Inline Error Feedback Summary

**One-liner:** Two-phase hotkey swap that registers new key before dropping old, with Settings.vue inline error reverting KeyCapture via prop watcher.

## What Was Built

### Task 1 — Atomic hotkey swap in hotkey.rs

Rewrote `update_hotkey` so it:
1. Gets the launcher window handle (returns Err if not found)
2. Calls `on_shortcut(hotkey.as_str(), handler)` to register the new key first
3. If `on_shortcut` fails, returns `Err("'{hotkey}' could not be registered: {e}")` immediately — old hotkey is still registered
4. If `on_shortcut` succeeds, unregisters the old hotkey (non-fatal on failure, logged with `eprintln!`)
5. Persists `settings.hotkey = hotkey` and returns `Ok(())`

The startup `register()` function is unchanged — it retains Alt+Space fallback logic for first boot.

Unit test `test_update_hotkey_err_format` documents the error string format that Settings.vue should expect.

### Task 2 — Inline hotkey error in Settings.vue

- Added `hotkeyError` ref (`ref<string | null>(null)`)
- Rewrote `onHotkeyChange` to:
  - Clear `hotkeyError` at the start of each attempt
  - On success: update `settings.value.hotkey` and call `saveSettings()`
  - On failure: set `hotkeyError` with the Rust error message + "still using {oldHotkey}", revert `settings.value.hotkey = oldHotkey` (which triggers KeyCapture's prop watcher to revert displayed key), skip `saveSettings()`
- Added `<p v-if="hotkeyError" class="hotkey-error">{{ hotkeyError }}</p>` after the Hotkey Row
- Added `.hotkey-error` CSS with `color: #ff453a`, token-based font/spacing
