<objective>
Fix two frontend gaps from UAT:

1. Hotkey error message (Gap 1): `onHotkeyChange` in Settings.vue captures `oldHotkey` from `settings.value.hotkey`, but KeyCapture emits `update:modelValue` before `change`, so by the time `onHotkeyChange` fires, `settings.value.hotkey` is already the new (failing) value — making the snapshot stale, the error message incoherent, and the revert a no-op. Fix: track the actually-registered hotkey in a separate `lastRegisteredHotkey` ref that is only updated on successful invoke.

2. KeyCapture defensive hardening (Gap 2 Vue side): The `@keydown` handler on the `.key-capture` div lacks a `.prevent` modifier, which is harmless for most keys but is a defensive gap. Add `@keydown.prevent` to the div so all key events are suppressed at the DOM level (complementing the Rust-side WM_SYSCOMMAND fix in Plan 04).

Purpose: Error feedback for failed hotkey registration is currently broken; the revert never actually reverts anything. Users are left with a silently corrupted state.
Output: Settings.vue with correct lastRegisteredHotkey tracking; KeyCapture.vue with @keydown.prevent.
</objective>

async function onHotkeyChange(hotkey: string) {
  const oldHotkey = lastRegisteredHotkey.value   // always the last SUCCESSFULLY registered key
  hotkeyError.value = null
  try {
    await invoke('update_hotkey', { hotkey })
    lastRegisteredHotkey.value = hotkey           // update only on success

    await saveSettings()
  } catch (e: any) {
    console.warn('Hotkey update failed:', e)
    const msg = typeof e === 'string' ? e : (e?.message ?? 'Could not register hotkey')
    hotkeyError.value = `${msg} — still using ${oldHotkey}`
    settings.value.hotkey = oldHotkey            // triggers KeyCapture prop watcher → reverts display
    // Do NOT call saveSettings() — backend hotkey is unchanged
  }
}
