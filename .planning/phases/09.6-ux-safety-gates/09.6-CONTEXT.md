# Phase 09.6: UX Safety Gates - Context

**Gathered:** 2026-03-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Two targeted safety improvements: (1) a confirmation gate before irreversible system commands (shutdown and restart) implemented as a Vue overlay inside the launcher; (2) an atomic two-phase hotkey swap in `hotkey.rs` so a failed registration leaves the old hotkey active instead of leaving no active hotkey.

</domain>

<decisions>
## Implementation Decisions

### Confirmation UI style
- Custom Vue overlay rendered inside the launcher window — consistent with existing context menu overlay pattern, no additional Tauri plugin
- Centered modal card with a warning tone (distinct from the context menu style to signal danger)
- Background: dimmed/dim overlay covers result list; content not interactive while confirmation is pending
- Confirm button uses accent color, Cancel is subdued
- Per-command message copy — each command has its own title and body text:
  - Shutdown: "Shut down Windows?" / "Your work will be lost."
  - Restart: "Restart Windows?" / "Your work will be lost."

### Confirmation keyboard behavior
- Escape = Cancel (close overlay, preserve launcher state)
- Enter = Confirm (execute command, hide launcher)
- Confirm button is focused by default when overlay opens

### Launcher visibility during confirmation
- Launcher stays visible and shows the overlay when confirmation is triggered
- Result list and input are dimmed and locked (not interactive) while overlay is open
- On confirm: hide launcher, execute command
- On cancel: close overlay, restore full launcher interactivity — query and selected result preserved exactly

### Gate scope
- Shutdown and Restart require confirmation
- Sleep and Lock do NOT require confirmation — both are trivially reversible

### Two-phase hotkey swap
- New sequence: register new hotkey first → on success, unregister old
- If new hotkey registration fails: return `Err`, leave old hotkey unchanged — no Alt+Space fallback at runtime
- If new hotkey registers but old unregister fails: treat as success (log the failure, don't roll back)
- Alt+Space fallback is startup-only behavior; runtime swap uses true rollback to known-working old key

### Hotkey failure feedback in Settings UI
- `update_hotkey` returns `Err` when new key fails to register (existing pattern)
- Settings.vue: on `Err`, revert the displayed hotkey field to the old (still-active) value
- Show inline error below the hotkey field: e.g. "Alt+F1 could not be registered — still using Alt+Space"
- No toast or warning banner needed — inline field state is sufficient

### Claude's Discretion
- Exact CSS styling for the confirmation card (padding, border-radius, shadow) — match existing design tokens
- Dim overlay opacity value
- Transition/animation for overlay appear/disappear

</decisions>

<specifics>
## Specific Ideas

- The confirmation overlay should feel visually distinct from the context menu — it signals a destructive, irreversible action rather than a navigation choice
- The confirm button is focused by default (Enter to confirm), but double-Enter from the search flow is not a concern because the overlay is triggered by Enter on a selected system command result — the user must press Enter a second deliberate time

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- Context menu overlay pattern (`App.vue`): `v-if` overlay div with `position: fixed`, `@mousedown.prevent` on buttons, `@click.stop` guards — directly reusable as the structural template for the confirmation overlay
- `run_system_command` (`system_commands.rs`): currently hides launcher immediately on entry — change: only hide on confirm, not on function entry
- `update_hotkey` (`hotkey.rs`): already returns `Err(String)` to the frontend when fallback occurs — same error channel works for rollback errors
- `Settings.vue:105`: already `await invoke('update_hotkey', { hotkey })` with error handling — extend to revert displayed key on error

### Established Patterns
- Overlay locking: context menu uses `@mousedown.prevent` to prevent focus-loss auto-hide race — confirmation overlay needs the same guard
- `@mousedown.stop` on interactive overlay elements to prevent Tauri drag-region interference (Phase 08 pattern)
- `isTauriContext` guard pattern for any new Tauri invoke calls

### Integration Points
- `App.vue`: add `confirmPending` reactive state + `pendingCommand` string; intercept Enter-on-system-command to show overlay instead of invoking directly
- `hotkey.rs` `update_hotkey`: reorder to register-new first, then unregister-old; remove runtime Alt+Space fallback (keep only at startup in `register()`)

</code_context>
