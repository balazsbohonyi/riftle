<objective>
Fix Gap 2 Rust side: Windows dispatches `WM_SYSCOMMAND SC_KEYMENU` (0xF100) before the DOM keydown event for Alt+Space. The OS system menu appears because the WebView2 window's default wndproc handles it. `e.preventDefault()` in JS is called too late.

Fix: subclass the settings window HWND using `SetWindowSubclass` to intercept `WM_SYSCOMMAND` messages where `wParam & 0xFFF0 == 0xF100` (SC_KEYMENU) and return 0 without forwarding to the original wndproc. All other messages pass through via `DefSubclassProc`.

This must run in the Tauri `setup` callback after the settings window is created, using the same HWND acquisition pattern already present for the launcher's DWM customization.

Purpose: Without this fix, users cannot capture Alt+Space as their hotkey in the Settings UI because the OS intercepts it.
Output: Settings window HWND is subclassed; SC_KEYMENU is swallowed; all other messages pass through normally.
</objective>

**Step B — lib.rs:**
After the existing DWM customization block for the launcher (around line 376), add a new `#[cfg(target_os = "windows")]` block that subclasses the settings window:

```rust
#[cfg(target_os = "windows")]
if let Some(settings_win) = app.get_webview_window("settings") {
    use windows::Win32::Foundation::{HWND, LRESULT, WPARAM, LPARAM};
    use windows::Win32::UI::Controls::{SetWindowSubclass, DefSubclassProc};

    unsafe extern "system" fn subclass_proc(
        hwnd: HWND,
        msg: u32,
        wparam: WPARAM,
        lparam: LPARAM,
        _subclass_id: usize,
        _ref_data: usize,
    ) -> LRESULT {
        const WM_SYSCOMMAND: u32 = 0x0112;
        const SC_KEYMENU: usize = 0xF100;
        if msg == WM_SYSCOMMAND && (wparam.0 & 0xFFF0) == SC_KEYMENU {
            // Swallow Alt key menu activation — prevents OS system menu from appearing
            // when user presses Alt+Space in the KeyCapture input
            return LRESULT(0);
        }
        DefSubclassProc(hwnd, msg, wparam, lparam)
    }

    match settings_win.hwnd() {
        Ok(raw_hwnd) => {
            let hwnd = HWND(raw_hwnd.0 as *mut std::ffi::c_void);
            unsafe {
                let _ = SetWindowSubclass(hwnd, Some(subclass_proc), 1, 0);
            }
        }
        Err(err) => {
            eprintln!("[startup] failed to acquire settings HWND for subclassing: {}", err);
        }
    }
}
```

Place this block immediately after the existing DWM block (after its closing brace, before `Ok(())`).

The subclass ID `1` is arbitrary but must be unique per window — using 1 for the settings window is fine since the launcher is not subclassed.

The `subclass_proc` is a `unsafe extern "system"` free function — it must be declared at module scope (not inside a closure or nested block) because `SetWindowSubclass` takes a raw function pointer. Declare it as a `fn` at the top level of lib.rs or inside the `#[cfg(target_os = "windows")]` block using a named `unsafe extern "system" fn`.

**Important:** Do NOT change any other part of lib.rs. The DWM block for the launcher remains unchanged.
