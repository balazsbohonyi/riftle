# Phase 09.5 Research: Backend Resilience

## Scope Snapshot

Phase 09.5 is a narrow hardening pass, not a backend redesign.

The roadmap scope is:
- replace panic-prone runtime `.lock().unwrap()` / `.hwnd().unwrap()` paths in `src-tauri/src/commands.rs`, `src-tauri/src/search.rs`, and `src-tauri/src/lib.rs`
- add backup-before-reset plus user-visible warnings for DB/settings silent reset paths in `src-tauri/src/db.rs` and `src-tauri/src/store.rs`

The planner should explicitly avoid expanding this phase into:
- broader `indexer.rs` unwrap cleanup
- general persistence error propagation everywhere
- a full notification system or settings-window redesign

## Current Architecture Constraints

- Data lives under `data_dir`, resolved by `src-tauri/src/paths.rs`:
  - portable: `<exe_dir>/data`
  - installed: `%APPDATA%\\riftle-launcher`
- DB path is `data_dir/launcher.db` in `src-tauri/src/lib.rs:103`
- settings path is `data_dir/settings.json` in `src-tauri/src/store.rs:106`
- The app already uses Tauri window events for cross-boundary coordination:
  - Rust emits `launcher-show` in `src-tauri/src/lib.rs:24` and `src-tauri/src/hotkey.rs:31`
  - Vue listens in `src/App.vue:371`
  - Settings emits `settings-changed` to the launcher in `src/Settings.vue:127`, `src/Settings.vue:145`, `src/Settings.vue:152`
- There is no existing toast/banner infrastructure. The smallest compatible pattern is:
  - Rust emits an app/window event
  - Vue listens and renders a small inline warning

Important startup constraint:
- `lib.rs` currently does `get_settings()` and then immediately `set_settings()` on startup (`src-tauri/src/lib.rs:109-110`)
- That means any fallback-to-defaults path in `store.rs` is effectively an overwrite path unless the implementation explicitly tracks whether defaults came from a recovery/reset flow

## Runtime Panic Sites

### `src-tauri/src/commands.rs`

Relevant runtime unwraps:
- `src-tauri/src/commands.rs:15` DB lock during `launch()` path lookup
- `src-tauri/src/commands.rs:50` DB lock during `launch()` launch-count increment
- `src-tauri/src/commands.rs:99` DB lock during `launch_elevated()` path lookup
- `src-tauri/src/commands.rs:145` DB lock during `launch_elevated()` launch-count increment

Safest handling consistent with the codebase:
- Recover poisoned mutexes with `lock().unwrap_or_else(|e| { eprintln!(...); e.into_inner() })`
- Do not fail the command for launch-count update failures; log and continue
- For path lookup failures, keep current behavior style:
  - log
  - return `Ok(())`
  - do not show a user-facing warning

Why this matches the repo:
- other runtime paths already prefer “log and continue” over user-facing hard failure
- `search.rs` already recovers poisoned `RwLock` with `into_inner()` (`src-tauri/src/search.rs:63`, `src-tauri/src/search.rs:175`)
- launch commands are intentionally forgiving today

### `src-tauri/src/search.rs`

Relevant runtime unwraps:
- `src-tauri/src/search.rs:48` DB lock during `init_search_index()`
- `src-tauri/src/search.rs:58` DB lock during `rebuild_index()`

Recommended handling:
- Same poison-recovery pattern as above for the DB mutex
- `init_search_index()`:
  - if DB query fails, log and initialize an empty search index
- `rebuild_index()`:
  - if DB query fails, log and keep the existing in-memory index instead of replacing it with an empty index

Planner note:
- `rebuild_index()` should stay non-fatal because it is called after launch and reindex flows
- keeping the old index is a better recovery than silently blanking search results

### `src-tauri/src/lib.rs`

Relevant runtime unwraps in phase scope:
- `src-tauri/src/lib.rs:209` tray double-click clears `pending_left_click`
- `src-tauri/src/lib.rs:210` tray double-click updates `suppress_left_click_until`
- `src-tauri/src/lib.rs:219` tray click reads `suppress_left_click_until`
- `src-tauri/src/lib.rs:227` tray click updates `pending_left_click`
- `src-tauri/src/lib.rs:233` delayed single-click worker reads `pending_left_click`
- `src-tauri/src/lib.rs:261` `launcher.hwnd().unwrap()` for DWM attribute setup

Recommended handling:
- tray timestamp mutexes:
  - recover poison with `into_inner()`
  - log once per failure site
  - overwrite the timestamp state and continue
- DWM handle:
  - replace `.hwnd().unwrap()` with `match launcher.hwnd()`
  - on failure, log and skip DWM border/corner customization

Why this is safe:
- tray mutexes guard only `Option<Instant>` bookkeeping, not durable data
- DWM setup is cosmetic only; failure should not block startup

### Out Of Scope But Easy To Accidentally Expand Into

There are still runtime `.lock().unwrap()` calls in `src-tauri/src/indexer.rs` (`131`, `149`, `164`, `318`), but Phase 09.5 as inserted in the roadmap does not call those out. The planner should decide explicitly whether to keep them out of scope.

## Backup-Before-Reset Strategy

## DB Failure Path

Current behavior:
- `src-tauri/src/db.rs:47-55` retries `try_init_db()`
- on failure it deletes `launcher.db`
- then it recreates it
- no backup, no warning

Recommended behavior:
1. Attempt `try_init_db(db_path)`.
2. If it succeeds, return normally.
3. If it fails and `launcher.db` does not exist, return the original error.
4. If it fails and `launcher.db` exists:
   - back it up to `launcher.db.bak` in the same directory
   - overwrite any existing `.bak`
   - only after backup succeeds, remove/replace the original and retry fresh init
5. If backup cannot be created, do not delete the original DB. Return the error.

File location and naming:
- original: `<data_dir>/launcher.db`
- backup: `<data_dir>/launcher.db.bak`

Overwrite behavior:
- overwrite the previous `.bak`
- no timestamped fan-out in 09.5; fixed filenames are already in roadmap wording and keep the phase small

Implementation detail that matters:
- prefer same-directory rename/move first, then fresh DB creation at the original path
- if rename is not viable in the exact implementation path, a copy-then-delete fallback is acceptable
- do not delete the original DB unless backup succeeded

User warning content:
- title: `Launch history was reset`
- message should name the backup path and explain that Riftle recreated a fresh DB

## Settings Failure Path

Current behavior:
- `src-tauri/src/store.rs:105-115` returns `Settings::default()` on malformed/load failure
- `lib.rs` immediately persists that result via `set_settings()` (`src-tauri/src/lib.rs:109-110`)
- this is effectively a silent overwrite path

Recommended behavior:
- distinguish three cases:
  - file missing: return defaults, no warning, normal first-run path
  - file exists but cannot be opened/parsed: backup to `settings.json.bak`, return defaults, emit warning
  - backup fails: do not silently overwrite the original file

File location and naming:
- original: `<data_dir>/settings.json`
- backup: `<data_dir>/settings.json.bak`

Overwrite behavior:
- overwrite the previous `.bak`

Planner-critical API implication:
- `get_settings()` probably should not stay a bare `Settings` return if the phase is implemented cleanly
- the startup flow needs to know whether returned defaults came from:
  - first run
  - recovery after malformed file
  - hard failure where overwriting is unsafe

The clean small-shape solution is a load outcome enum/struct, for example:
- `Loaded(Settings)`
- `RecoveredWithDefaults { settings: Settings, warning: BackendWarning }`
- `Fatal(String)`

That keeps recovery logic in `store.rs` but lets `lib.rs` decide whether to call `set_settings()`.

## Shared Backup Helper Recommendation

Both `db.rs` and `store.rs` want the same file operation semantics:
- same-directory `.bak`
- overwrite existing backup
- do not destroy original unless backup succeeded

The planner should consider a tiny shared helper module instead of duplicating file logic in both files. A small `backup_file_with_overwrite(path, bak_suffix)` helper is enough.

## Warning Surfacing: Minimum Viable Integration

The smallest warning architecture that fits the app is:

Backend:
- add a small warning payload type, for example:
  - `kind`: `db-reset` | `settings-reset`
  - `title`
  - `message`
  - `backup_path`
- add a managed in-memory queue for startup warnings and missed events
- emit a Tauri event when a warning occurs
- add one command to drain pending warnings after mount

Frontend:
- implement warning rendering only in `src/App.vue`
- on mount:
  - call the drain command once to catch startup warnings
  - start listening for the event for later warnings
- render a simple dismissible inline banner above the search box

Why this is the right minimum:
- no new dependency
- no modal dialog
- no toast system
- no Settings UI changes required
- startup warnings are not lost if they fire before Vue listeners are attached

Recommended event/command shape:
- event: `backend-warning`
- command: `take_backend_warnings`

What not to do in 09.5:
- do not add a full notification center
- do not wire warnings into both windows unless it falls out cheaply
- do not block startup on user acknowledgement

## Test Opportunities

## Automated

Best-value unit tests:
- `db.rs`
  - corrupt/invalid existing DB triggers `launcher.db.bak` creation
  - existing `.bak` is overwritten
  - missing DB file does not create backup
  - backup failure does not delete the original DB
- `store.rs`
  - malformed settings file triggers `settings.json.bak`
  - existing `.bak` is overwritten
  - missing settings file is treated as first run, not warning
  - recovery outcome correctly signals whether defaults should be persisted
- shared helper
  - backup overwrite semantics
  - original preserved on backup failure
- unwrap hardening helpers, if extracted
  - poisoned mutex recovery returns inner value without panic

Good practical testing strategy:
- extract small pure helpers for backup and recovery decisions
- avoid trying to unit test full Tauri window/event plumbing deeply

## Manual / Smoke

Manual validation should cover:
- corrupt `settings.json`, start app, verify:
  - `settings.json.bak` exists in the active data dir
  - launcher shows a warning banner on first summon
  - defaults load
- corrupt `launcher.db`, start app, verify:
  - `launcher.db.bak` exists in the active data dir
  - app starts with fresh index/history
  - warning banner appears
- tray click/double-click still works after lock recovery changes
- launcher still starts if DWM hwnd acquisition fails or is skipped

## Risks

- `src-tauri/src/lib.rs` is already a cross-phase hotspot and currently has local modifications in the worktree; planner should keep edits surgical
- startup timing is the main integration risk: warnings emitted too early will be lost unless there is a pending-warning queue
- `store.rs` uses `tauri-plugin-store`, but backup needs raw-file semantics; planner should avoid relying only on deserialized store values
- the current `get_settings()` + immediate `set_settings()` startup pattern is the easiest place to reintroduce silent overwrite regressions
- if backup creation fails, the safe behavior is to avoid destructive reset even if that means startup returns an error

## Validation Architecture

The validation doc for 09.5 should be built around four independently checkable claims:

1. Runtime panic sites are now recoverable.
- Verify the listed `commands.rs`, `search.rs`, and `lib.rs` unwrap sites no longer use crash-on-poison / crash-on-handle-failure behavior.
- Code review target: fallible handling is explicit and logged.

2. DB destructive reset is backup-first and user-visible.
- Verify `launcher.db.bak` is created before reset.
- Verify the app surfaces a warning payload/banner.
- Verify prior DB contents are not silently destroyed without backup.

3. Settings destructive recovery is backup-first and user-visible.
- Verify `settings.json.bak` is created before defaults are persisted after a malformed-file recovery.
- Verify first-run missing-file behavior does not emit a false warning.

4. Frontend integration stayed minimal.
- Verify one warning surface exists in the launcher.
- Verify startup warnings are visible after mount.
- Verify no unnecessary UI subsystems were introduced.

Suggested validation evidence:
- unit test results for backup helpers and recovery outcomes
- one manual smoke for corrupted DB
- one manual smoke for corrupted settings
- one quick runtime smoke for tray/launcher behavior after unwrap replacements

## Planner Guidance Summary

The planner should optimize for:
- a tiny shared warning/backup helper layer
- startup-safe warning delivery
- explicit recovery outcomes for settings load
- file-local unwrap hardening with poison recovery, not broad refactors

The two places most likely to produce bad plans are:
- forgetting that `get_settings()` is immediately followed by `set_settings()` at startup
- using event-only warning delivery and losing startup warnings before Vue listeners mount
