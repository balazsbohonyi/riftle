## Summary

This phase fixes four concrete, pre-identified bugs in the Rust backend. All bugs are fully understood from direct source inspection — no external research is needed. All four fixes are isolated to two files (`indexer.rs` and `store.rs`) and the changes are small and surgical.

**Bug 1** (`reindex` uses `Settings::default()`): Line 226 of `indexer.rs` ignores live settings. The `reindex()` command already receives `app: tauri::AppHandle` and `data_dir_state: tauri::State<PathBuf>`, so `crate::store::get_settings(&app, &data_dir)` is a one-line fix.

**Bug 2** (timer interval not updated): The `start_background_tasks` function captures `interval_mins` as a copied `u32` on the stack. Changing settings later has no effect on the running thread. The fix is to upgrade the existing `mpsc::Sender<()>` to a message enum (`TimerMsg::Reset` / `TimerMsg::SetInterval(u32)`) so `set_settings_cmd` can push a new interval into the running thread.

**Bug 3** (`interval_mins=0` causes continuous indexing): `Duration::from_secs(0)` means `deadline = Instant::now() + Duration::ZERO`, which is always in the past. The fix is a guard at the top of the timer loop: when `interval_mins == 0`, skip the deadline check entirely.

**Bug 4** (`system_tool_allowlist` missing from `get_settings_cmd`): The manually-built `serde_json::json!({})` in `get_settings_cmd` (lines 143–155, `store.rs`) simply omits `system_tool_allowlist`. Add the field. The `set_settings_cmd` deserialises via `serde(default)`, so omitting the field means the allowlist is silently reset to defaults on every save.

**Primary recommendation:** Fix all four bugs in a single focused pass on `indexer.rs` and `store.rs`, with the timer interval upgrade as the most structurally complex change. Write tests first (TDD).

## Architecture Patterns

### Pattern 1: Load live settings inside `reindex()` (SIC-01)

**What:** Replace `crate::store::Settings::default()` with an actual `get_settings()` call.

**Where:** `src-tauri/src/indexer.rs`, inside the `reindex()` Tauri command, the closure passed to `std::thread::spawn`.

**Key constraint:** `get_settings()` takes `&AppHandle` and `&Path`. The `reindex()` command receives `app: tauri::AppHandle` which is `Clone`. The spawned thread needs ownership — clone before moving into the closure.

**Pattern:**
```rust
// Inside the std::thread::spawn closure in reindex():
let data_dir = data_dir_state.inner().clone();
let app_clone = app.clone();  // needed inside thread
std::thread::spawn(move || {
    if flag.compare_exchange(...).is_ok() {
        let settings = crate::store::get_settings(&app_clone, &data_dir);
        run_full_index(&db, &data_dir, &settings);
        // ...
    }
});
```

**Note:** `tauri::AppHandle` implements `Clone` — confirmed by Tauri v2 API design and existing usage in `lib.rs` (e.g., the tray handler clones `app.handle().clone()`).

### Pattern 2: Timer interval update via message enum (SIC-02 + SIC-03)

**What:** Replace `mpsc::Sender<()>` with `mpsc::Sender<TimerMsg>` where `TimerMsg` is:

```rust
enum TimerMsg {
    Reset,              // manual reindex completed — reset deadline to now+interval
    SetInterval(u32),   // settings changed — use new interval (0 = disabled)
}
```

**Where:** Defined in `indexer.rs`. The `start_background_tasks` function signature changes from returning `mpsc::Sender<()>` to `mpsc::Sender<TimerMsg>`. The managed state type in `lib.rs` changes from `Arc<Mutex<mpsc::Sender<()>>>` to `Arc<Mutex<mpsc::Sender<TimerMsg>>>`.

**Timer loop logic (new):**
```rust
let mut interval_mins = settings.reindex_interval;  // mutable local
let mut deadline = if interval_mins == 0 {
    None  // disabled
} else {
    Some(Instant::now() + Duration::from_secs(interval_mins as u64 * 60))
};

loop {
    std::thread::sleep(Duration::from_secs(1));
    match timer_rx.try_recv() {
        Ok(TimerMsg::Reset) => {
            if interval_mins > 0 {
                deadline = Some(Instant::now() + Duration::from_secs(interval_mins as u64 * 60));
            }
        }
        Ok(TimerMsg::SetInterval(new_interval)) => {
            interval_mins = new_interval;
            deadline = if new_interval == 0 {
                None
            } else {
                Some(Instant::now() + Duration::from_secs(new_interval as u64 * 60))
            };
        }
        Err(TryRecvError::Disconnected) => break,
        Err(TryRecvError::Empty) => {}
    }
    if let Some(dl) = deadline {
        if Instant::now() >= dl {
            try_start_index(&is_indexing, &db, &data_dir, &settings_clone);
            deadline = Some(Instant::now() + Duration::from_secs(interval_mins as u64 * 60));
        }
    }
    // If deadline is None (interval_mins == 0) — skip entirely
}
```

**Where `SetInterval` is sent:** `set_settings_cmd` in `store.rs` needs access to the timer sender. Options:

- **Option A (recommended):** Have `set_settings_cmd` accept `timer_tx: tauri::State<Arc<Mutex<mpsc::Sender<TimerMsg>>>>` as an additional parameter, then send `TimerMsg::SetInterval(settings.reindex_interval)` after persisting. This is consistent with how `reindex()` already uses `timer_tx: tauri::State<...>`.
- **Option B:** Emit a backend event after settings save; a listener in `lib.rs` reads from managed state. More indirection — not recommended.

Option A is the correct pattern for this codebase.

**Impact on `lib.rs`:** The `app.manage(Arc::new(Mutex::new(timer_tx)))` line needs the type updated. The managed state is `Arc<Mutex<mpsc::Sender<TimerMsg>>>`. This is a type-only change — no logic change in `lib.rs`.

**Impact on `reindex()` command:** `timer_tx.lock().unwrap().send(())` becomes `timer_tx.lock().unwrap().send(TimerMsg::Reset)`.

### Pattern 3: Add `system_tool_allowlist` to `get_settings_cmd` (SIC-04)

**What:** One line addition to the `serde_json::json!({})` block.

**Where:** `src-tauri/src/store.rs`, `get_settings_cmd` function, lines 143–155.

**Before:**
```rust
serde_json::json!({
    "hotkey": settings.hotkey,
    // ... other fields ...
    "animation": settings.animation,
    "data_dir": data_dir.to_string_lossy(),
    "is_portable": is_portable,
})
```

**After:** Add:
```rust
    "system_tool_allowlist": settings.system_tool_allowlist,
```

This is the minimal correct fix. The `Settings` struct already has `#[serde(default = "default_system_tool_allowlist")]` so partial JSON omitting this field would deserialise to the hardcoded default — not to the user's saved value.

### Pattern 4: Test strategy (SIC-05)

**Constraint:** Tests must be in-process Rust unit tests in the same files where the logic lives. No `AppHandle` available in `#[test]` context.

**What is testable without `AppHandle`:**

| Test | Location | Approach |
|------|----------|----------|
| `get_settings_cmd` JSON includes `system_tool_allowlist` | `store.rs` | Construct a `Settings` with non-default allowlist, call `serde_json::json!()` directly with the same expression, assert key present |
| Serde round-trip preserves `system_tool_allowlist` | `store.rs` | `serde_json::to_value(&settings)` → `serde_json::from_value::<Settings>()` → compare allowlist |
| `interval_mins=0` does not trigger deadline | `indexer.rs` | Unit-test the timer logic in isolation: compute `deadline = if interval_mins == 0 { None } else { Some(...) }`, assert `deadline.is_none()` |
| `TimerMsg::Reset` re-arms deadline when interval > 0 | `indexer.rs` | Same isolated logic test |
| `reindex` calls `get_settings` (not default) | `indexer.rs` | Extract the settings-loading logic into a testable helper, or test `get_index_paths` with non-default `additional_paths` to confirm they appear |

**What requires manual smoke testing:**
- The actual `reindex()` Tauri command (requires `AppHandle` + managed state)
- Timer thread behavior over time (real-time, impractical for unit tests)
- `get_settings_cmd` end-to-end (requires `AppHandle` + `tauri::State`)

**Test placement:**
- Timer deadline logic tests → `indexer.rs` `#[cfg(test)] mod tests`
- Settings round-trip + JSON contract tests → `store.rs` `#[cfg(test)] mod tests`

### Anti-Patterns to Avoid

- **Do not add `AppHandle` as a type parameter to `get_index_paths` or `crawl_dir`** — these are pure functions testable without Tauri context. Keep the separation: `reindex()` command fetches settings, then passes them to the pure functions.
- **Do not store `Settings` as managed state** — the project decision (STATE.md line 198) is full-struct replace, not partial patch. Managed state for settings would complicate the set/get pattern unnecessarily.
- **Do not use `Arc<Mutex<Settings>>` for the timer** — the `TimerMsg::SetInterval(u32)` approach requires only transmitting a single u32 (the interval). It avoids cloning the full settings struct across threads for what is a simple scalar update.
- **Do not change `try_start_index` signature** — it already takes `&Settings` from its caller. The timer loop just needs to track `interval_mins` locally and pass the current settings clone (for `additional_paths`, etc.) separately. Since the timer's settings snapshot is from startup, and SIC-01 fixes the manual reindex path, the timer's stale settings are a separate future concern (not in scope for this phase per the bug descriptions).

## Common Pitfalls

### Pitfall 1: Cloning `AppHandle` before moving into thread

**What goes wrong:** `app: tauri::AppHandle` is moved into the `std::thread::spawn` closure. If you try to use `app` after the `spawn` call (e.g., in `crate::search::rebuild_index(&app)`), the compiler rejects it.

**Why it happens:** `reindex()` already calls `rebuild_index(&app)` after the thread completes — but the thread must own its own `AppHandle` clone.

**How to avoid:** Clone `app` before spawning: `let app_for_thread = app.clone();` and move the clone into the closure.

**Warning signs:** `use of moved value: app` compiler error.

### Pitfall 2: `TimerMsg` type must be `Send`

**What goes wrong:** `mpsc::Sender<TimerMsg>` requires `TimerMsg: Send`. A `TimerMsg` enum containing only `u32` and unit variants is automatically `Send`.

**Why it happens:** Cross-thread channels require `Send`.

**How to avoid:** Keep `TimerMsg` variants to primitive types (`u32`, `()`). No `Arc`, no raw pointers.

### Pitfall 3: `set_settings_cmd` parameter count change breaks frontend if not updated

**What goes wrong:** `set_settings_cmd` currently takes `(app, data_dir, settings)`. Adding `timer_tx` as a 4th `tauri::State<...>` parameter is purely a Rust-side change — Tauri extracts `State<T>` parameters from managed state automatically, not from the frontend `invoke()` call. Frontend does not need to change.

**Why it happens:** Tauri v2 distinguishes between `State<T>` parameters (injected from managed state) and regular parameters (from frontend JSON). `State<T>` parameters are invisible to the frontend.

**How to avoid:** Understand that adding `tauri::State<Arc<Mutex<mpsc::Sender<TimerMsg>>>>` to `set_settings_cmd` does not change the frontend `invoke('set_settings_cmd', { settings: ... })` call.

**Warning signs:** If you think the frontend needs to change — stop, it does not.

### Pitfall 4: Managed state type mismatch at runtime

**What goes wrong:** `lib.rs` manages `Arc<Mutex<mpsc::Sender<()>>>`. After changing to `TimerMsg`, if `lib.rs` is not updated, `tauri::State<Arc<Mutex<mpsc::Sender<TimerMsg>>>>` panics at runtime with "no state registered for type".

**Why it happens:** Tauri matches managed state by type identity. Old and new types are distinct.

**How to avoid:** Update `app.manage(Arc::new(Mutex::new(timer_tx)))` in `lib.rs` at the same time as the `indexer.rs` type change. The type is inferred from `timer_tx` — if `start_background_tasks` returns `mpsc::Sender<TimerMsg>`, the `manage()` call automatically wraps the right type.

**Warning signs:** Runtime panic in Tauri state resolution (only caught at runtime, not compile time).

### Pitfall 5: `interval_mins=0` edge case in `Reset` handling

**What goes wrong:** After a manual reindex, the timer sends `TimerMsg::Reset` which re-arms the deadline. If `interval_mins` is currently 0 (manual only), the `Reset` handler must NOT set a deadline.

**How to avoid:** Guard the `Reset` arm: `if interval_mins > 0 { deadline = Some(...); }` — same as shown in the pattern above.

    Reset,
    SetInterval(u32),

let mut deadline: Option<Instant> = if interval_mins == 0 {
    None
} else {
    Some(Instant::now() + Duration::from_secs(interval_mins as u64 * 60))

loop {
    std::thread::sleep(Duration::from_secs(1));
    match timer_rx.try_recv() {
        Ok(TimerMsg::Reset) => {
            if interval_mins > 0 {
                deadline = Some(Instant::now() + Duration::from_secs(interval_mins as u64 * 60));
            }
        }
        Ok(TimerMsg::SetInterval(n)) => {
            interval_mins = n;
            deadline = if n == 0 {
                None

                Some(Instant::now() + Duration::from_secs(n as u64 * 60))

        Err(TryRecvError::Empty) => {}

## Open Questions

1. **Where to define `TimerMsg`**
   - What we know: it is used in `indexer.rs` (defined + sent by timer thread), `store.rs` (sends `SetInterval`), and `lib.rs` (managed state type)
   - What's unclear: whether to define in `indexer.rs` and import into `store.rs`, or move to a shared module
   - Recommendation: define `pub enum TimerMsg` in `indexer.rs` (where the timer lives), use `crate::indexer::TimerMsg` in `store.rs`. No new module needed.

2. **Should `set_settings_cmd` take `Option<State<...>>` for the timer sender?**
   - What we know: `timer_tx` is only managed in `#[cfg(desktop)]` block; `set_settings_cmd` may be called in any context
   - What's unclear: whether Tauri panics if a `State<T>` parameter is referenced but `T` was never managed
   - Recommendation: Use `app.try_state::<Arc<Mutex<mpsc::Sender<TimerMsg>>>>()` inside the command body (returns `Option`) instead of a `tauri::State<...>` parameter — safe for both desktop and non-desktop builds.
