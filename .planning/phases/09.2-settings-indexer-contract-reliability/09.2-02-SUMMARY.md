# Phase 09.2 Plan 02: Timer Loop Rewrite + reindex() Live Settings Summary

**TimerMsg enum with Option<Instant> deadline timer loop (None = disabled when interval=0) and reindex() wired to live settings via get_settings() instead of Settings::default()**

## Accomplishments

- Added `pub enum TimerMsg { Reset, SetInterval(u32) }` to indexer.rs — typed channel replaces unit `()` sender
- Rewrote timer thread body to use `Option<Instant>` deadline: `None` when `interval_mins == 0` (timer disabled), `Some(...)` otherwise — eliminates the Duration::from_secs(0) always-past bug
- Fixed `reindex()` command to call `crate::store::get_settings(&app_for_thread, &data_dir)` so manual re-index respects user-configured additional_paths and excluded_paths
- `lib.rs` type inference correctly picks up `Arc<Mutex<mpsc::Sender<TimerMsg>>>` with zero changes — confirmed by zero-error build
- All 43 tests green; all 4 Wave 0 contract tests (zero_interval, nonzero_interval, set_interval, live_settings) confirmed passing

## Files Created/Modified

## Decisions Made

- `lib.rs` `app.manage(Arc::new(Mutex::new(timer_tx)))` infers `Arc<Mutex<mpsc::Sender<TimerMsg>>>` from `start_background_tasks` return type — no explicit annotation required. The compiler verified this without any `let timer_sender: Arc<Mutex<...>> = ...` annotation.
- The timer thread `settings` variable (cloned from `start_background_tasks` parameter) is used for `try_start_index` calls. The `interval_mins` is separately tracked as `mut` local and updated by `SetInterval`. This means try_start_index uses startup-time settings for crawl config; live settings for crawl config come through `reindex()` only. Plan 04 will wire `set_settings_cmd` to send `SetInterval` for the interval; crawl config (additional_paths, excluded_paths) is always live through the `reindex()` path.
- Tasks 1 and 2 committed atomically in a single commit because Task 2 required zero file changes.
