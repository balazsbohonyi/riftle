# Phase 09.10: Settings with Sidebar - Context

**Gathered:** 2026-06-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Reorganize the existing Settings window into a sidebar-based layout. The phase changes Settings navigation and section layout only: existing settings controls, persistence, validation, and live update behavior should continue to work the same way unless explicitly noted below.

</domain>

<decisions>
## Implementation Decisions

### Sidebar Layout
- Add a left sidebar listing each Settings section title.
- Clicking a sidebar section title shows that section's settings in the main content area.
- The content area shows one section at a time, not all sections stacked vertically.
- Section order is fixed as: General, Appearance, Search, Shortcuts.
- Settings always opens with General selected.
- The Settings window should be made wider as needed to fit the sidebar without crowding the content area.
- Sidebar section titles must each fit on a single line; do not wrap section labels.
- Use `--color-bg-darker` for the sidebar background.
- The active sidebar row should use `--color-selection-bg` as its selected background, matching the selected entry style in the launcher search results.
- Dark and light themes both need to look intentional and readable with the new sidebar.

### Section Organization
- Remove the separate Hotkey section.
- Move the existing "Global shortcut" setting into the General section.
- "Global shortcut" should be the first row in General, before any other General settings.
- Keep all other existing section settings functionally unchanged.
- Sidebar sections should reflect the reorganized sections, so the standalone Hotkey item should not appear.

### Shortcut Scroll Behavior
- Remove the special shortcut wrapper scroll handling that was added to avoid two simultaneous scrollbars.
- With one section displayed in the content area at a time, the Shortcuts section no longer needs the fixed wrapper behavior that suppresses nested scrolling for directory/file shortcut entries.
- Let the active content area be the only scroll container; shortcut lists should grow naturally inside the Shortcuts section.
- Directory and file shortcut editing should otherwise work exactly as it does now.

### Functional Preservation
- No behavioral changes are required for settings persistence, immediate save behavior, hotkey registration, theme switching, path editing, reindexing, appearance toggles, or shortcut validation.
- Existing controls should keep their current labels and interactions unless they must move to fit the new layout.
- Existing error states, including hotkey registration errors and shortcut validation errors, should remain visible in the appropriate active section.

### Claude's Discretion
- Exact sidebar width, window width, spacing, hover styling, and content padding, as long as labels do not wrap and the layout looks polished in both themes.
- Whether the active content section keeps the existing `Section.vue` title treatment or uses a lighter variant appropriate for single-section display.
- Exact implementation shape for active section state in `Settings.vue`.

</decisions>

<specifics>
## Specific Ideas

- The sidebar is for section titles only; it is not a nested settings tree.
- The requested regrouping is specifically: put "Global shortcut" in General before "Launch at startup" and "Play sound on open".
- The special scroll case to remove is tied to the current Shortcuts wrapper around directory and file shortcut lists.
- The active sidebar row should visually align with the launcher's selected search result treatment by using `--color-selection-bg`.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/Settings.vue`: Current Settings implementation. It renders all sections in one scrollable `.settings-content` area and owns settings loading, saving, Tauri event emission, hotkey updates, shortcut validation, and reindex actions.
- `src/components/ui/Section.vue`: Existing section wrapper with title, divider, optional description, and body slot. Can be reused for the active content section.
- `src/components/ui/Row.vue`: Existing row layout for label/control pairs. Continue using it for moved settings.
- `src/components/ui/KeyCapture.vue`: Existing "Global shortcut" control. Move it into General without changing behavior.
- `src/components/ui/ShortcutList.vue`: Existing directory/file shortcut editor with its own hidden scrollbar and custom thumb. The planner should review whether the component-level scroll remains necessary after removing the outer fixed panel special case.
- `src/styles/tokens.css`: Defines `--color-bg-darker` for both default/dark and light themes.

### Established Patterns
- Settings use Vue 3 `<script setup lang="ts">` with refs and local handlers.
- Settings save immediately by sending the full settings object through `set_settings_cmd`.
- Launcher-visible settings updates are propagated with `emitTo('launcher', 'settings-changed', ...)`.
- Tauri access is guarded by `isTauriContext`; browser dev mode should keep working.
- Design uses CSS custom properties from `tokens.css`, not hardcoded theme-specific colors.

### Integration Points
- `src/Settings.vue` template: replace the single stacked `.settings-content` section list with a two-column shell containing the sidebar and the active section content area.
- `src/Settings.vue` state: add active section state for sidebar navigation.
- `src/Settings.vue` styles: add sidebar styling, widen the window layout as needed, and remove the fixed `.shortcut-tab-panel` height / nested shortcut list height overrides that supported the old double-scrollbar workaround.
- Settings window sizing may require updates in `src-tauri/tauri.conf.json` if the configured width/min width is too narrow for the sidebar layout.

</code_context>
