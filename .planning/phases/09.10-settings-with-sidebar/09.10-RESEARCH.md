## User Constraints (from CONTEXT.md)

### Locked Decisions

### Sidebar Layout
- Add a left sidebar listing each Settings section title.
- Clicking a sidebar section title shows that section's settings in the main content area.
- The content area shows one section at a time, not all sections stacked vertically.
- Section order is fixed as: General, Appearance, Search, Shortcuts.
- Settings always opens with General selected.
- The Settings window should be made wider as needed to fit the sidebar without crowding the content area.
- Sidebar section titles must each fit on a single line; do not wrap section labels.
- Use `--color-bg-darker` for the sidebar background.
- The active sidebar row should use `--color-selection-bg` as its selected background, matching the selected entry style in the launcher search results.
- Dark and light themes both need to look intentional and readable with the new sidebar.

### Section Organization
- Remove the separate Hotkey section.
- Move the existing "Global shortcut" setting into the General section.
- "Global shortcut" should be the first row in General, before any other General settings.
- Keep all other existing section settings functionally unchanged.
- Sidebar sections should reflect the reorganized sections, so the standalone Hotkey item should not appear.

### Shortcut Scroll Behavior
- Remove the special shortcut wrapper scroll handling that was added to avoid two simultaneous scrollbars.
- With one section displayed in the content area at a time, the Shortcuts section no longer needs the fixed wrapper behavior that suppresses nested scrolling for directory/file shortcut entries.
- Let the active content area be the only scroll container; shortcut lists should grow naturally inside the Shortcuts section.
- Directory and file shortcut editing should otherwise work exactly as it does now.

### Functional Preservation
- No behavioral changes are required for settings persistence, immediate save behavior, hotkey registration, theme switching, path editing, reindexing, appearance toggles, or shortcut validation.
- Existing controls should keep their current labels and interactions unless they must move to fit the new layout.
- Existing error states, including hotkey registration errors and shortcut validation errors, should remain visible in the appropriate active section.

### Claude's Discretion
- Exact sidebar width, window width, spacing, hover styling, and content padding, as long as labels do not wrap and the layout looks polished in both themes.
- Whether the active content section keeps the existing `Section.vue` title treatment or uses a lighter variant appropriate for single-section display.
- Exact implementation shape for active section state in `Settings.vue`.

### Specific Ideas

- The sidebar is for section titles only; it is not a nested settings tree.
- The requested regrouping is specifically: put "Global shortcut" in General before "Launch at startup" and "Play sound on open".
- The special scroll case to remove is tied to the current Shortcuts wrapper around directory and file shortcut lists.
- The active sidebar row should visually align with the launcher's selected search result treatment by using `--color-selection-bg`.

None - discussion stayed within phase scope.

## Summary

This phase is a scoped Vue/CSS reorganization of the existing Settings window. The current `src/Settings.vue` owns the data model, persistence, Tauri IPC calls, hotkey rebinding, shortcut validation, custom settings scrollbar, and all section rendering. The planner should keep those behaviors in place and restructure only the presentation: add a sidebar, introduce active section state, move the hotkey row into General, remove the Hotkey section, and remove the fixed Shortcuts wrapper height rules.

No new library is warranted. The standard stack is the existing Vue 3.5 + Tauri 2 setup, existing UI primitives (`Section`, `Row`, `Toggle`, `KeyCapture`, `PathList`, `ShortcutList`, `Dropdown`, `Button`), and CSS design tokens from `src/styles/tokens.css`. Official Vue guidance supports class/style object bindings for active state and `v-show` for frequent toggles where DOM state should remain mounted; W3C APG confirms this is a tabbed-interface pattern when one panel is visible at a time.

**Primary recommendation:** Implement this as a single `Settings.vue` layout/state refactor plus a settings window width update in `src-tauri/tauri.conf.json`; do not touch Rust settings behavior or add routing/state libraries.

### Pattern 1: Section Registry + Local Active State

**What:** Define a typed list of sidebar sections and a local `activeSection` ref defaulting to `general`.

**When to use:** Use for the sidebar buttons and for deciding which section panel is visible. Keep the list fixed in the required order: General, Appearance, Search, Shortcuts.

**Example:**
```typescript
type SettingsSectionId = 'general' | 'appearance' | 'search' | 'shortcuts'

const activeSection = ref<SettingsSectionId>('general')

const settingsSections: Array<{ id: SettingsSectionId; label: string }> = [
  { id: 'general', label: 'General' },
  { id: 'appearance', label: 'Appearance' },
  { id: 'search', label: 'Search' },
  { id: 'shortcuts', label: 'Shortcuts' },
]
```

**Why:** Vue's official class binding docs support object/array bindings for reactive active classes. Source: https://vuejs.org/guide/essentials/class-and-style

### Pattern 2: Sidebar as Vertical Tabs

**What:** Treat the sidebar as a tab list: buttons select one associated content panel. Use `aria-selected`, `aria-controls`, and `role="tab"`/`role="tabpanel"` when practical.

**When to use:** Use because only one panel is visible and each sidebar row is a section label. W3C APG describes tabs as layered content sections where one panel is displayed at a time.

**Example:**
```vue
<aside class="settings-sidebar" role="tablist" aria-orientation="vertical" aria-label="Settings sections">
  <button
    v-for="section in settingsSections"
    :id="`settings-tab-${section.id}`"
    :key="section.id"
    type="button"
    role="tab"
    :aria-selected="activeSection === section.id"
    :aria-controls="`settings-panel-${section.id}`"
    :class="['settings-sidebar-item', { 'settings-sidebar-item--active': activeSection === section.id }]"
    @click="activeSection = section.id"
  >
    {{ section.label }}
  </button>
</aside>
```

**Source:** https://www.w3.org/WAI/ARIA/apg/patterns/tabs/

### Pattern 3: Preserve Panel State with `v-show`

**What:** Use `v-show` for section panels so only the active section is visible while child component state stays mounted.

**When to use:** Use especially for Shortcuts, because `ShortcutList.vue` has internal draft/edit/error/scroll state. Destroying it on every sidebar click can discard unsaved in-progress edits.

**Example:**
```vue
<main ref="settingsContentRef" class="settings-content" @scroll="onSettingsScroll">
  <Section
    id="settings-panel-general"
    role="tabpanel"
    aria-labelledby="settings-tab-general"
    v-show="activeSection === 'general'"
    title="General"
  >
    <Row label="Global shortcut" ref="hotkeyRowRef">
      <!-- existing hotkey row unchanged -->
    </Row>
    <Row label="Launch at startup" :hint="autostartHint">
      <!-- existing autostart toggle unchanged -->
    </Row>
    <Row label="Play sound on open">
      <!-- existing play sound toggle unchanged -->
    </Row>
  </Section>
</main>
```

**Source:** Vue documents `v-show` as keeping elements rendered while toggling CSS display, and recommends it when toggling often. https://vuejs.org/guide/essentials/conditional

### Pattern 4: Token-Driven Sidebar Styling

**What:** Add a two-column settings body with fixed sidebar width and flexible content. Sidebar uses `--color-bg-darker`; active row uses `--color-selection-bg`.

**When to use:** Use for both light and dark themes. Existing tokens define `--color-bg-darker` and `--color-selection-bg` for default, explicit light, and system-light modes.

**Example:**
```css
.settings-body {
  display: grid;
  grid-template-columns: 140px minmax(0, 1fr);
  min-height: 0;
  flex: 1;
}

.settings-sidebar {
  background: var(--color-bg-darker);
  border-right: 1px solid var(--color-divider);
  padding: var(--spacing-sm);
}

.settings-sidebar-item {
  width: 100%;
  white-space: nowrap;
}

.settings-sidebar-item--active {
  background: var(--color-selection-bg);
  color: var(--color-selection-text);
}
```

**Local sources:** `src/styles/tokens.css` lines 5, 17, 55, 67, 90, 102; launcher selection uses `--color-selection-bg` in `src/App.vue` around the selection bar.

### Anti-Patterns to Avoid

- **Adding Vue Router:** This is a four-section settings panel inside one Tauri window, not route-level navigation.
- **Moving settings logic to Rust:** The phase explicitly preserves behavior; Rust settings commands are already working.
- **Using `v-if` for every section without checking state loss:** It may destroy `ShortcutList` form state when the user switches sections.
- **Leaving `.shortcut-tab-panel` fixed at 340px:** This preserves the old double-scroll workaround and violates the active-content-only scroll decision.
- **Hardcoding dark/light colors:** Use tokens; both themes must remain readable.
- **Using wrapped sidebar labels:** Context explicitly requires single-line sidebar section titles.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Settings navigation | Router, nested navigation tree, custom page system | Local `activeSection` ref and fixed section registry | Four static panels; smaller and easier to verify. |
| Active row styling | New palette or ad hoc selected color | `--color-selection-bg` and `--color-selection-text` | Required by context and already theme-aware. |
| Global shortcut input | New hotkey capture UI | Existing `KeyCapture.vue` + existing `onHotkeyChange` | Existing code handles capture lifecycle, reset, rollback, and backend registration errors. |
| Row/control layout | Inline one-off flex rows | Existing `Row.vue`, `Toggle.vue`, `Dropdown.vue`, `Button.vue`, `PathList.vue`, `ShortcutList.vue` | Keeps visual and behavioral consistency. |
| Window resizing logic | Runtime JS resize loop | `tauri.conf.json` width/minWidth update | Settings window is non-resizable and static; Tauri config supports width/minWidth. |
| Shortcut validation | Duplicate frontend validators | Existing `validateShortcutSettings()` and `ShortcutList` validation | Avoids changing shortcut behavior in a visual reorganization phase. |

**Key insight:** The hard part is preserving stateful controls while changing layout. Most regressions would come from deleting or remounting working controls, not from missing a new library.

## Common Pitfalls

### Pitfall 1: Losing Hotkey Error Visibility

**What goes wrong:** Startup hotkey conflict opens Settings, but the error is hidden because the sidebar defaults to another section or the row ref moved incorrectly.

**Why it happens:** Current code scrolls `hotkeyRowRef` into view after `get_startup_hotkey_conflict`; moving the row can break that ref or hide it behind a non-General active section.

**How to avoid:** Keep `activeSection` defaulting to `general`, move `ref="hotkeyRowRef"` with the "Global shortcut" row, and keep `hotkeyError` immediately under that row.

**Warning signs:** Startup conflict opens Settings without visible error text; build passes but manual smoke fails.

### Pitfall 2: Accidentally Changing Save Semantics

**What goes wrong:** Settings stop saving immediately, or hotkey failure gets persisted.

**Why it happens:** Moving template blocks can accidentally switch handlers or call `saveSettings()` from the wrong path.

**How to avoid:** Reuse the existing handlers unchanged: `onHotkeyChange`, `onAutostartChange`, `onThemeChange`, `onPathsChange`, `onShortcutsChange`, and the `emitTo('launcher', 'settings-changed', ...)` calls.

**Warning signs:** `pnpm build` catches no error, but toggles no longer update launcher or failed hotkey shows stale UI.

### Pitfall 3: Double Scrollbars Remain

**What goes wrong:** Shortcuts still show a fixed 340px inner panel and a separate list scroll/thumb.

**Why it happens:** The old rules are at `.shortcut-tab-panel`, `.shortcut-tab-panel :deep(.shortcut-list)`, and `.shortcut-tab-panel :deep(.shortcut-list-shell)`.

**How to avoid:** Remove fixed height/overflow rules in `Settings.vue`. Let `.settings-content` be the only constrained scroll container. `ShortcutList` can keep its component styles if it grows naturally when unconstrained.

**Warning signs:** In Shortcuts, the list area scrolls independently while the sidebar/content shell also scrolls.

### Pitfall 4: Sidebar Crowds Existing Row Controls

**What goes wrong:** At the current 450px settings window width, sidebar plus content compresses rows, dropdowns, and KeyCapture.

**Why it happens:** Current settings config is `width: 450` and `minWidth: 450`; a 120-150px sidebar leaves too little content width.

**How to avoid:** Increase settings window `width` and `minWidth` in `src-tauri/tauri.conf.json`. Tauri v2 config documents `width`, `height`, `minWidth`, and `resizable`. A practical target is 640-700px wide while keeping 600px height.

**Warning signs:** "Global shortcut" row wraps, sidebar labels wrap, or dropdown controls squeeze.

### Pitfall 5: ARIA Semantics Without Keyboard Behavior

**What goes wrong:** Sidebar uses `role="tab"` but keyboard users cannot move predictably.

**Why it happens:** ARIA roles describe behavior; they do not implement behavior.

**How to avoid:** Either use plain buttons with clear active styling, or if using tab roles, implement at least click, Tab reachability, and optionally Up/Down/Home/End per W3C APG. Because this is a desktop utility and the scope is visual reorganization, plain buttons are acceptable if keyboard focus remains usable.

**Warning signs:** Focus disappears, Tab order skips the active panel, or roles are present without matching focus behavior.

## Open Questions

1. **Should the sidebar implement full WAI-ARIA tab keyboard navigation?**
   - What we know: The layout matches W3C's tabs pattern because one panel is shown at a time.
   - What's unclear: The user did not explicitly request arrow-key navigation in Settings sidebar.
   - Recommendation: Use real `<button>` elements and preserve Tab accessibility. Add `role="tablist"` semantics only if the plan also includes basic Up/Down/Home/End behavior.

2. **Exact settings window width**
   - What we know: Existing 450px is too narrow for sidebar plus content.
   - What's unclear: Final visual width preference.
   - Recommendation: Plan for 660px width/minWidth, then verify visually in Tauri dev. Adjust within Claude's discretion if content still crowds.

3. **Whether to change `Section.vue` title styling**
   - What we know: Existing section title is uppercase/accent and can be reused.
   - What's unclear: Whether single-section display should look lighter.
   - Recommendation: Reuse `Section.vue` first. Only adjust styling if visual verification shows the title treatment feels too heavy.
