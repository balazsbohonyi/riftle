# Phase 09.7: Shortcuts Support - Context

**Gathered:** 2026-05-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Add user-configured shortcuts as first-class launcher results. Shortcuts are either directories or files. They are configured in a new Shortcuts section in Settings, below Search, with separate Directories and Files subsections. Directory shortcuts open folders. File shortcuts open files with their associated default application, and executable/app file shortcuts may include launch parameters.

This phase does not add a plugin system, web search, shell command runner, or arbitrary app workflow automation.

</domain>

<decisions>
## Implementation Decisions

### Matching and Ranking
- Shortcut matching is prefix-first. Prefix matches on alias or fallback name should be treated as the primary shortcut match behavior.
- Matching shortcuts always appear above normal app search results.
- If alias is empty, directory shortcuts use the directory basename, and file shortcuts use the filename without extension as the fallback display and match name.
- Duplicate aliases are blocked in Settings before save.
- For executable/app file shortcuts with parameters, alias is mandatory. This avoids ambiguous parameterized shortcuts whose fallback name could collide with the base executable or file name.

### Settings Editing
- Settings gains a separate Shortcuts section below Search.
- The Shortcuts section has two subsections: Directories and Files.
- Directory shortcut entries expose path plus optional alias.
- File shortcut entries expose path, optional parameters for executable/app targets, then alias as the last field.
- Adding uses native pickers: folder picker for directory shortcuts, file picker for file shortcuts.
- Validation happens before save. Invalid entries are rejected in the Settings UI, including duplicate aliases, missing alias for parameterized executable/app shortcuts, and parameters on non-executable/non-app files.

### Launch Behavior
- Directory shortcuts open the directory through the OS default folder handling, then hide the launcher on success.
- Non-executable file shortcuts open through the OS default file association.
- If a non-executable file has no usable association, show the Windows Open With flow if possible.
- Executable/app shortcuts with parameters pass the parameter text as a raw arguments string. The user controls quoting.
- If a shortcut target is missing or launching fails, keep the launcher open and show a lightweight warning so the user can fix the shortcut in Settings.

### Icons and Display
- Directory shortcuts use a generic directory icon.
- File shortcuts use a generic file icon by default.
- If possible, extract an icon for executable/app shortcuts.
- If possible, extract the icon of the associated app for non-executable file shortcuts; this is desirable but can fall back to the generic file icon.
- Result rows should use the existing launcher result style. When path display is enabled or disambiguation is needed, show the target path as secondary text.

### Claude's Discretion
- Exact inline layout for shortcut entry editing, as long as it stays consistent with existing Settings rows and compact control density.
- Exact warning copy for failed shortcut launch.
- Exact icon filenames and fallback hierarchy, provided icon access remains constrained to app-owned icon assets.

</decisions>

<specifics>
## Specific Ideas

- The main parameter use case is launching a project path through an executable, for example a shortcut that runs VS Code with a project path argument.
- Alias should be the final field in the setup flow for file shortcuts.
- Parameter support is mainly for apps and executable files, not directories or ordinary document files.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/components/ui/PathList.vue`: existing add/remove path list pattern, but it only supports folder paths. Shortcuts likely need a richer sibling component rather than overloading this one.
- `src/components/ui/Section.vue`, `Row.vue`, `Button.vue`, `Dropdown.vue`: existing Settings primitives should be reused for the new Shortcuts section.
- `src-tauri/src/indexer.rs::extract_icon_png`: existing Windows icon extraction path for executables can inform executable/app shortcut icon handling.
- `src-tauri/src/commands.rs::get_icon_bytes`: existing safe icon-byte IPC path should be extended carefully rather than returning arbitrary filesystem paths to the frontend.

### Established Patterns
- Settings are persisted as a full `Settings` struct in `src-tauri/src/store.rs`; any shortcut data should round-trip through `get_settings_cmd` and `set_settings_cmd`.
- OS interactions belong in Rust. Vue should call IPC commands for shortcut launching, icon loading, validation results, and settings persistence.
- Search results use `SearchResult { id, name, icon_path, path, kind, requires_elevation }`. Shortcut results should fit this result-row contract, likely with a distinct `kind`.
- Launcher result interaction is keyboard-first: Enter launches selected result, Ctrl+Shift+Enter currently routes to elevated launch for non-system results.

### Integration Points
- `src/Settings.vue`: add the Shortcuts section below Search and wire changes through `saveSettings()`.
- `src-tauri/src/store.rs`: add shortcut settings data structures with serde defaults and JSON command round-trip coverage.
- `src-tauri/src/search.rs`: merge shortcut matches ahead of normal app results and preserve the 50-result cap.
- `src/App.vue`: route shortcut results to a shortcut launch IPC command rather than normal `launch(id)`.
- `src-tauri/src/commands.rs`: add shortcut launch behavior for folders, files, executables with parameters, and launch failure reporting.

</code_context>

<deferred>
## Deferred Ideas

- Binding a directory shortcut to a specific application is out of scope for this phase.
- A structured argument/template builder is out of scope; raw argument strings are sufficient for this phase.

</deferred>
