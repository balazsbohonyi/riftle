## User Constraints (from CONTEXT.md)

### Locked Decisions
## Implementation Decisions

### Matching and Ranking
- Shortcut matching is prefix-first. Prefix matches on alias or fallback name should be treated as the primary shortcut match behavior.
- Matching shortcuts always appear above normal app search results.
- If alias is empty, directory shortcuts use the directory basename, and file shortcuts use the filename without extension as the fallback display and match name.
- Duplicate aliases are blocked in Settings before save.
- For executable/app file shortcuts with parameters, alias is mandatory. This avoids ambiguous parameterized shortcuts whose fallback name could collide with the base executable or file name.

### Settings Editing
- Settings gains a separate Shortcuts section below Search.
- The Shortcuts section has two subsections: Directories and Files.
- Directory shortcut entries expose path plus optional alias.
- File shortcut entries expose path, optional parameters for executable/app targets, then alias as the last field.
- Adding uses native pickers: folder picker for directory shortcuts, file picker for file shortcuts.
- Validation happens before save. Invalid entries are rejected in the Settings UI, including duplicate aliases, missing alias for parameterized executable/app shortcuts, and parameters on non-executable/non-app files.

### Launch Behavior
- Directory shortcuts open the directory through the OS default folder handling, then hide the launcher on success.
- Non-executable file shortcuts open through the OS default file association.
- If a non-executable file has no usable association, show the Windows Open With flow if possible.
- Executable/app shortcuts with parameters pass the parameter text as a raw arguments string. The user controls quoting.
- If a shortcut target is missing or launching fails, keep the launcher open and show a lightweight warning so the user can fix the shortcut in Settings.

### Icons and Display
- Directory shortcuts use a generic directory icon.
- File shortcuts use a generic file icon by default.
- If possible, extract an icon for executable/app shortcuts.
- If possible, extract the icon of the associated app for non-executable file shortcuts; this is desirable but can fall back to the generic file icon.
- Result rows should use the existing launcher result style. When path display is enabled or disambiguation is needed, show the target path as secondary text.

### Claude's Discretion
- Exact inline layout for shortcut entry editing, as long as it stays consistent with existing Settings rows and compact control density.
- Exact warning copy for failed shortcut launch.
- Exact icon filenames and fallback hierarchy, provided icon access remains constrained to app-owned icon assets.

## Summary

Implement shortcuts as settings-backed, first-class search results, not as indexed `apps` rows. This keeps user-configured shortcuts out of the crawler/prune lifecycle, preserves the existing SQLite app index, and matches the phase boundary: shortcuts are user settings, not discovered applications.

Use the existing `Settings` full-struct round trip in `store.rs`, extend `SearchResult` with shortcut result kinds, and merge shortcut matches before normal app results inside `search.rs`. Launching remains Rust-owned: `App.vue` should dispatch `kind === "shortcut_dir" | "shortcut_file"` to a new `launch_shortcut(id)` command, and `commands.rs` should use `ShellExecuteW` for folder/file/app launch plus `SHOpenWithDialog` only for `SE_ERR_NOASSOC` on non-executable files.

**Primary recommendation:** Add `directory_shortcuts` and `file_shortcuts` to `Settings`, derive validated shortcut search results in `search.rs`, and launch by stable `shortcut:<kind>:<hash>` IDs resolved from current settings at launch time.

### Pattern 1: Settings-Backed Shortcut Model
**What:** Add typed shortcut arrays to `Settings`.
**When to use:** Always for this phase; shortcuts are manually configured user data.
**Example:**
```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub struct DirectoryShortcut {
    pub path: String,
    #[serde(default)]
    pub alias: String,
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub struct FileShortcut {
    pub path: String,
    #[serde(default)]
    pub parameters: String,
    #[serde(default)]
    pub alias: String,
}

#[serde(default)]
pub directory_shortcuts: Vec<DirectoryShortcut>,
#[serde(default)]
pub file_shortcuts: Vec<FileShortcut>,
```

### Pattern 2: Prefix-First Shortcut Merge
**What:** Compute shortcut matches from current settings and prepend them before app results.
**When to use:** Inside `search(query, index_state, app, data_dir)` after `>` system-command branch.
**Example:**
```rust
let mut shortcuts = search_shortcuts(&query, &settings);
let remaining = 50usize.saturating_sub(shortcuts.len());
let mut apps = score_and_rank(&query, &index.apps);
apps.truncate(remaining);
shortcuts.extend(apps);
shortcuts
```

### Pattern 3: Rust-Owned Launch Dispatch
**What:** `App.vue` routes shortcut results to Rust; Rust validates target existence and executes.
**When to use:** For both directory and file shortcuts.
**Example:**
```typescript
if (item.kind === 'shortcut_dir' || item.kind === 'shortcut_file') {
  const launched = await invoke<boolean>('launch_shortcut', { id: item.id })
  if (launched) await hideWindow()
  return
}
```

### Anti-Patterns to Avoid
- **Indexing shortcuts into `apps`:** `prune_stale()` would remove missing targets and blur user shortcuts with discovered apps.
- **Launching shortcut paths from Vue:** violates the Rust-owned OS boundary and makes Windows-specific failure handling harder.
- **Passing parameters for document files:** Microsoft documents `lpParameters` for executable files; document files should pass `NULL`.
- **Returning arbitrary icon paths to Vue:** Phase 09.3 hardened icon loading; keep `icon_path` as validated app-owned filenames served by `get_icon_bytes`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Folder/file launch | Custom `cmd /c start` wrapper | `ShellExecuteW` with default/open verb | Handles shell verbs, folders, file associations, and Windows UI correctly. |
| Open With fallback | `rundll32 shell32.dll,OpenAs_RunDLL` | `SHOpenWithDialog` + `OPENASINFO` | `SHOpenWithDialog` is the supported API; `OpenAs_RunDLL` is undocumented. |
| File/folder pickers | Custom input browsing | `@tauri-apps/plugin-dialog` `open()` | Already installed, native UI, already permitted. |
| Argument parser | Tokenizer/quoting builder | Raw `lpParameters` string | User explicitly controls quoting; structured builder is deferred. |
| Icon file serving | Expose filesystem paths | Existing `get_icon_bytes` with validated filenames | Prevents path traversal and respects Phase 09.3 asset hardening. |

**Key insight:** the hard parts here are Windows shell semantics and preserving existing security boundaries. Use OS APIs and existing app-owned icon plumbing rather than inventing command execution, picker, or asset-serving layers.

## Common Pitfalls

### Pitfall 1: Hiding The Launcher Before A Failed Shortcut Launch
**What goes wrong:** Missing target or association failure hides the launcher, leaving no visible feedback.
**Why it happens:** Existing `launch(id)` hides before `ShellExecuteW` for perceived performance.
**How to avoid:** `launch_shortcut` should return `Ok(false)` or a structured warning on failure; Vue hides only after success.
**Warning signs:** shortcut launch failures only appear in stderr, no visible warning in launcher.

### Pitfall 2: Duplicate Aliases Across Shortcut Types
**What goes wrong:** `work` could match both a directory and a file shortcut, making prefix-first ordering unpredictable.
**Why it happens:** validating directory aliases and file aliases separately.
**How to avoid:** normalize alias/fallback names case-insensitively across both arrays before save.
**Warning signs:** same alias appears in Settings under both Directories and Files.

### Pitfall 3: Empty Alias With Parameterized Executable
**What goes wrong:** `code.exe` with parameters appears as `code`, colliding with indexed VS Code or another shortcut.
**Why it happens:** fallback naming hides the semantic difference between executable and parameterized invocation.
**How to avoid:** reject any file shortcut with non-empty `parameters` and empty alias before save.
**Warning signs:** Settings permits parameters while alias field is blank.

### Pitfall 4: Treating Every File With Parameters As Valid
**What goes wrong:** `ShellExecuteW` receives document path plus `lpParameters`, which Microsoft says should be `NULL` for document files.
**Why it happens:** UI validation only checks whether parameter text exists.
**How to avoid:** allow parameters only for a conservative executable extension allowlist, preferably `.exe`, `.com`, `.bat`, `.cmd`; document `.lnk` parameter behavior as out of scope unless planner explicitly validates it.
**Warning signs:** parameters accepted for `.pdf`, `.txt`, `.docx`, images, or directories.

### Pitfall 5: Shortcut Icons Breaking Asset Hardening
**What goes wrong:** frontend receives `C:\...` or user-controlled icon paths.
**Why it happens:** associated-app icon extraction returns a filesystem path or icon reference string.
**How to avoid:** write any extracted icon into `{data_dir}/icons/{hash}.png`, return only `generic.png`, `shortcut_folder.png`, `shortcut_file.png`, or validated 16-hex PNG filenames.
**Warning signs:** `icon_path` contains slash, backslash, drive letters, or non-`.png` extensions.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `rundll32 shell32.dll,OpenAs_RunDLL` | `SHOpenWithDialog` | Windows Vista+, current docs updated 2024-02-22 | Use supported API for Open With fallback. |
| Asset protocol `asset://` with broad scope | Rust `get_icon_bytes` + validated filenames | Phase 09.3 | Shortcut icons must use the same app-owned icon pipeline. |
| Static settings snapshot for reindex/timer | Live settings read and timer update messages | Phase 09.2 | Shortcut search should read current settings or update a managed shortcut cache on save. |

**Deprecated/outdated:**
- `OpenAs_RunDLL`: useful folklore but undocumented; use `SHOpenWithDialog`.
- Global asset protocol paths: removed by Phase 09.3 hardening; do not reintroduce.

## Open Questions

1. **Should parameterized `.lnk` shortcuts be supported in Phase 09.7?**
   - What we know: existing indexer launches `.lnk` files because ShellExecute preserves stored shortcut arguments; Microsoft documents `lpParameters` for executable files, not documents/shortcuts.
   - What's unclear: whether user expects adding extra parameters to `.lnk` targets.
   - Recommendation: Phase 09.7 should support raw parameters for `.exe`, `.com`, `.bat`, `.cmd` only; `.lnk` launches as-is unless a follow-up explicitly expands target resolution.

2. **Should launch counts apply to shortcuts?**
   - What we know: shortcuts always sort above apps and context did not request MRU ranking for shortcuts.
   - What's unclear: whether users expect frequently used shortcuts to sort within shortcut matches.
   - Recommendation: skip shortcut launch_count in this phase; sort exact/prefix quality first, then stable Settings order.
