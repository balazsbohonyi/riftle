# Phase 09.7 Plan 04: Shortcut User Experience Summary

## Accomplishments

- Added `ShortcutList.vue` for directory and file shortcut editing with add, edit, save, cancel, remove, path browsing, alias handling, and file parameters.
- Added the Settings `Shortcuts` section with Directories and Files tabs, compact read-only rows, inline validation, persistence through the existing settings round-trip, and explanatory copy for Search versus Shortcuts paths.
- Wired launcher search results with `kind: "shortcut_dir"` and `kind: "shortcut_file"` through `launch_shortcut`, hiding the launcher only after successful launches and leaving warnings visible on failures.
- Fixed Windows document shortcut launch behavior by using Tauri opener for default associated app flows and avoiding accidental activation of app windows instead of opening the file.
- Added native-looking shortcut icons: folder icons for directories and associated/explicit app icons for file shortcuts, with cache/database reuse where available.
- Refined the shortcut settings layout into a tabbed, compact, hover-action UI with overlay scrollbars to prevent width jumpiness.
- Adjusted launcher context menu placement so it opens below the cursor instead of under it.
- Completed UAT for eight user-facing shortcut checks with all tests passing.

## Decisions Made

- New shortcut forms appear inline at the top of each tab after the native picker is used.
- Existing entries remain text-first until the user explicitly clicks edit.
- Empty file parameters are hidden in read-only entries; non-empty parameters are displayed as a compact single line with tooltip truncation.
- Removing a shortcut happens immediately.
- Invalid edit paths are rejected before saving.
- Scrollbars overlay the content instead of reserving width, avoiding tab and settings window layout shifts.

## Deviations from Plan

- Plan 04 originally described Directories and Files as subsections under Shortcuts. User feedback changed this to equal-width tabs with embedded add buttons.
- Runtime testing exposed Windows shell behavior where opening file shortcuts could switch virtual desktops or activate the associated app instead of opening the file. The implementation was adjusted to use Tauri opener for default file association behavior.
- The icon implementation was broadened beyond generic placeholder icons to use folder and associated/executable app icons.
- Additional explanatory text was added to Search and Shortcuts to distinguish indexed app paths from user-created shortcut results.

## Issues Encountered

- Windows document shortcut launching initially changed virtual desktops or focused the associated app instead of opening the target file. Fixed by using `tauri-plugin-opener` and `openPath` semantics.
- Shortcut row and settings scrollbars initially caused content width jumpiness. Fixed with overlay scrollbars shown only while scrolling.
- Shortcut entries initially used the generic Tauri icon. Fixed with native directory and app-associated icon handling.
- Context menu initially appeared under the cursor near the Quit action. Fixed by moving it below the pointer.
