# Phase 09.4 Plan 03: Indexer Hardening Wave 2 Summary

**Rayon-bounded icon pool (4 threads) and dedicated COM worker thread isolating all .lnk resolution through a balanced CoInitializeEx/CoUninitialize lifetime**

## Accomplishments

- Added rayon = "^1" and replaced unbounded `std::thread::spawn` for icon extraction with a `rayon::ThreadPool` capped at 4 threads, preventing GDI call storms during large re-indexes
- Implemented `spawn_com_worker()` — a dedicated long-lived thread that owns the COM apartment; `CoInitializeEx` is called once on start, `CoUninitialize` on exit; `resolve_lnk` no longer calls `CoInitializeEx`
- Wired `com_tx: SyncSender<LnkQuery>` through the entire indexer pipeline: `crawl_dir`, `run_full_index`, `try_start_index`, `start_background_tasks`, `reindex()` Tauri command, and `lib.rs` startup
- All 59 Rust unit tests pass (0 failures, 2 ignored); test_com_tx() helper added so tests exercise the COM worker path

## Decisions Made

- rayon ThreadPool Drop blocks run_full_index exit (implicit join) — same observable behavior as thread::spawn but bounds concurrency to 4 threads
- Per-request allowlist in LnkQuery: baking the allowlist into spawn_com_worker at startup would make stale allowlists possible after settings changes; passing it per-request is the correct design
- S_FALSE (already initialized) result from CoInitializeEx in resolve_lnk is now impossible — COM is guaranteed initialized on the worker thread before resolve_lnk runs; removing the inline call makes the contract explicit
