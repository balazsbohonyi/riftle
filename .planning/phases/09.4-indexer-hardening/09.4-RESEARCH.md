# Phase 09.4: Indexer Hardening - Research

**Researched:** 2026-03-11
**Domain:** Rust / Windows — thread pool concurrency, WalkDir traversal, Windows COM lifecycle, Win32 path limits
**Confidence:** HIGH

---

## Summary

Phase 09.4 addresses five discrete robustness concerns in `indexer.rs`. All changes are confined to a single file and its callers. Each item targets a different failure mode: resource exhaustion (item 6), silent exclusion bypass (item 9), runaway traversal (item 10), COM lifetime hazard (item 11), and path truncation (item 12).

The changes do not alter any public API surface visible to the frontend or other modules. They are internal implementation improvements. Two items (6 and 11) require adding a new crate dependency (`rayon`). Three items (9, 10, 12) require only stdlib and existing crate changes.

The most structurally significant change is item 11 (COM isolation), which requires funneling all `resolve_lnk` calls through a dedicated long-lived thread via an `mpsc` channel rather than calling the function inline in the crawl loop. This serializes `.lnk` resolution but eliminates the unbalanced `CoInitializeEx` risk. Item 6 (thread pool) is the other structural change — replacing per-item `thread::spawn` calls with `rayon::ThreadPool::scope`.

**Primary recommendation:** Add `rayon = "1"` to Cargo.toml. Implement items in this order: 9 (path normalization, zero-risk), 10 (WalkDir guards, zero-risk), 6 (bounded pool, low-risk), 12 (extended path buffer, low-risk), 11 (COM isolation, medium structural complexity).

## Architecture Patterns

### Recommended Structure After This Phase

No directory changes. All modifications are within `src-tauri/src/indexer.rs` plus one `Cargo.toml` addition.

The key structural change is the introduction of two long-lived objects:
- `icon_pool: rayon::ThreadPool` — created once in `run_full_index`, used for icon extraction scope
- COM worker thread — created once in `run_full_index` or `start_background_tasks`, receives `.lnk` paths via channel, returns resolved targets via reply channel

### Pattern 1: Bounded Thread Pool with rayon::scope (Item 6)

**What:** Replace `std::thread::spawn` in the icon extraction loop with `rayon::ThreadPool::scope`, using a pool bounded to `num_cpus` (default rayon behavior) or a hard cap like 4.

**When to use:** When N independent tasks need to be completed before proceeding, with bounded parallelism.

**Current code (the problem):**
```rust
// indexer.rs line 85 — one thread per uncached icon, unbounded
std::thread::spawn(move || {
    match extract_icon_png(&exe_path) { ... }
});
```

**Pattern:**
```rust
// Source: https://docs.rs/rayon/latest/rayon/struct.ThreadPoolBuilder.html
let icon_pool = rayon::ThreadPoolBuilder::new()
    .num_threads(4)  // cap at 4 concurrent GDI calls
    .build()
    .unwrap_or_else(|_| rayon::ThreadPoolBuilder::new().build().unwrap());

// Collect apps needing icon extraction, then process with bounded pool
// rayon::scope blocks until all spawned tasks finish
icon_pool.scope(|s| {
    for (exe_path, app_id, icon_file, db_clone) in pending_icons {
        s.spawn(move |_| {
            // same body as current thread::spawn closure
        });
    }
});
```

**Important note:** `rayon::scope` blocks the caller until all spawned tasks complete. Since `run_full_index` is called from a spawned thread (in `reindex()`) and from setup (before app is shown), blocking is acceptable. The current approach also effectively blocks in that all threads need to finish before `prune_stale` runs — but with unbounded `thread::spawn` there is no join point. The scope makes the join explicit.

**Alternative: fire-and-forget with rayon::spawn (no blocking)**

If non-blocking icon extraction is desired (matching current "fire and forget" behavior):

```rust
// For true fire-and-forget with bounded pool, use pool.spawn() without scope
// This does NOT wait for all tasks to finish
icon_pool.spawn(move || { ... });
```

However, `icon_pool` must then be kept alive (stored as managed state or in an Arc). The `scope` approach is simpler and the blocking behavior matches user expectation (re-index completes fully before rebuilding search index).

### Pattern 2: Path Normalization for Exclusion (Item 9)

**What:** Before calling `path.starts_with(ex)`, normalize both `path` and `ex` to lowercase absolute paths with consistent separators.

**When to use:** Any time user-supplied path strings are compared against filesystem paths.

**Current code (the problem):**
```rust
// indexer.rs line 322 — raw String comparison, case-sensitive, separator-sensitive
if excluded.iter().any(|ex| path.starts_with(ex)) {
```

**Pattern:**
```rust
// Source: std::path::Path docs — canonicalize + fallback
fn normalize_path_for_comparison(p: &Path) -> String {
    // canonicalize resolves symlinks, normalizes separators, resolves . and ..
    // Falls back to lossy lowercase if path doesn't exist (excluded paths may not exist)
    p.canonicalize()
        .unwrap_or_else(|_| p.to_path_buf())
        .to_string_lossy()
        .to_lowercase()
}

// In crawl_dir: pre-normalize excluded list once, not per-entry
let normalized_excluded: Vec<String> = excluded
    .iter()
    .map(|ex| normalize_path_for_comparison(Path::new(ex)))
    .collect();

// Then per entry:
let norm_path = normalize_path_for_comparison(path);
if normalized_excluded.iter().any(|ex| norm_path.starts_with(ex.as_str())) {
    continue;
}
```

**Important:** Pre-normalize the excluded list once outside the per-file loop (not once per WalkDir entry) for performance.

**Windows-specific:** `Path::canonicalize` on Windows resolves to the canonical case returned by the filesystem, which is case-preserving but case-insensitive. The `.to_lowercase()` call is still needed because `starts_with` is case-sensitive in Rust.

### Pattern 3: WalkDir Depth and Symlink Guards (Item 10)

**What:** Add `max_depth` to WalkDir and change `follow_links(true)` to `follow_links(false)` (the WalkDir default). `follow_root_links` remains `true` (the default) so root paths that are themselves symlinks are still traversed.

**When to use:** Any recursive directory traversal where user-supplied paths could be symlink cycles or very deep trees.

**Current code (the problem):**
```rust
// indexer.rs line 316 — no depth limit, follows all symlinks
let walker = walkdir::WalkDir::new(root).follow_links(true);
```

**Pattern:**
```rust
// Source: https://docs.rs/walkdir/latest/walkdir/struct.WalkDir.html
let walker = walkdir::WalkDir::new(root)
    .max_depth(8)           // 8 levels deep is ample for Start Menu structure
    .follow_links(false);   // do not follow symlinks encountered during traversal
// follow_root_links defaults to true — root symlinks are still traversed
```

**Depth selection rationale:** Windows Start Menu directories typically have 3-4 levels (Programs / Publisher / App). Desktop is flat (depth 1). Additional paths are user-chosen and could be deep. 8 is a reasonable cap that covers real-world app installations while preventing runaway on NTFS junction point loops.

**Note:** `same_file_system(true)` is another guard option but is redundant when `follow_links(false)` is set.

### Pattern 4: COM Worker Thread Isolation (Item 11)

**What:** Instead of calling `resolve_lnk` (which calls `CoInitializeEx`) inline during the crawl loop, send each `.lnk` path to a dedicated long-lived thread via an `mpsc` channel. That thread calls `CoInitializeEx` once at startup and `CoUninitialize` before exit, and processes all resolution requests serially.

**When to use:** Any Windows COM usage that must have a clear init/uninit lifecycle.

**Why the current approach is risky:**

From official COM docs: "each successful call to CoInitializeEx... must be balanced by a corresponding call to CoUninitialize." The current code calls `CoInitializeEx` once per `resolve_lnk` invocation (one per `.lnk` file found) and never calls `CoUninitialize`. COM reference-counts init calls per thread. If `resolve_lnk` is called N times on the same thread, COM is initialized N times on that thread but uninitialized 0 times. On thread exit, the OS will reclaim resources, but COM objects still alive may not release cleanly. More seriously, if the rayon worker pool (item 6) or WalkDir iteration runs `resolve_lnk` on rayon-managed threads, COM is initialized per-rayon-thread with no cleanup path.

**Channel-based COM worker pattern:**
```rust
// Source: COM docs https://learn.microsoft.com/en-us/windows/win32/api/combaseapi/nf-combaseapi-coinitializeex
// + std::sync::mpsc (stdlib)

struct LnkRequest {
    lnk_path: PathBuf,
    reply_tx: std::sync::mpsc::SyncSender<Option<PathBuf>>,
}

fn start_com_worker(allowlist: Vec<String>) -> std::sync::mpsc::SyncSender<LnkRequest> {
    let (tx, rx) = std::sync::mpsc::sync_channel::<LnkRequest>(64);
    std::thread::spawn(move || {
        use windows::Win32::System::Com::{CoInitializeEx, CoUninitialize, COINIT_APARTMENTTHREADED};
        unsafe {
            let _ = CoInitializeEx(None, COINIT_APARTMENTTHREADED);
        }
        for req in rx {
            let result = resolve_lnk_inner(&req.lnk_path, &allowlist);
            let _ = req.reply_tx.send(result);
        }
        // rx dropped (channel closed) -> loop exits -> CoUninitialize called
        unsafe { CoUninitialize(); }
    });
    tx
}
```

**Usage in crawl_dir:**
```rust
// Synchronous: send request, block on reply
let (reply_tx, reply_rx) = std::sync::mpsc::sync_channel(1);
let _ = com_tx.send(LnkRequest { lnk_path: path.to_path_buf(), reply_tx });
if let Ok(Some(target)) = reply_rx.recv() {
    apps.push(make_app_record(&target, source, name));
}
```

**Performance note:** Serializing `.lnk` resolution through one thread is slower than parallel resolution. However, `.lnk` resolution is I/O-bound (reading the shortcut file + checking target existence) and the COM IShellLinkW apartment model (COINIT_APARTMENTTHREADED) serializes COM calls anyway. In practice the bottleneck is icon extraction, not `.lnk` resolution. Serialization is acceptable.

**Alternative (simpler, lower risk):** Call `CoUninitialize` at the end of `resolve_lnk` if and only if `CoInitializeEx` returned `S_OK` (not `S_FALSE`). Track the return value of `CoInitializeEx` and conditionally uninit. This avoids the architectural change but is subtler and still runs COM on arbitrary threads. The dedicated thread approach is architecturally cleaner.

### Pattern 5: Extended-Length Path Buffer for .lnk Resolution (Item 12)

**What:** Replace the `MAX_PATH` (260 `u16`) buffer in `GetPath` with a 32,767 `u16` buffer (the Windows extended-path maximum).

**When to use:** Any Win32 API call that takes a character buffer for a path.

**Current code (the problem):**
```rust
// indexer.rs line 389 — MAX_PATH = 260 characters
let mut buf = vec![0u16; MAX_PATH as usize];
```

**The constraint:** `IShellLinkW::GetPath` documentation states "The maximum path size that can be returned is `MAX_PATH`." This is a documented API limitation — the COM interface itself is bounded by `MAX_PATH` regardless of buffer size. However, using a larger buffer (32,767) is still correct practice:

1. It future-proofs against any Windows update relaxing this constraint
2. It prevents buffer overrun if an implementation ever returns more than 260 chars
3. It aligns with the `\\?\` extended-path convention

**Pattern:**
```rust
// Source: https://learn.microsoft.com/en-us/windows/win32/fileio/maximum-file-path-limitation
// Extended path limit is 32,767 characters
const EXTENDED_MAX_PATH: usize = 32_767;
let mut buf = vec![0u16; EXTENDED_MAX_PATH];
```

**Realistic impact:** For `IShellLinkW::GetPath` specifically, the API documentation says MAX_PATH is the ceiling. The main benefit is the `\\?\` prefix pattern: if a `.lnk` target has a path that was stored with the extended-length prefix, the 32,767 buffer handles it correctly. The 260-char truncation silently drops such paths — they fail with an empty buffer and return `None`, which is the current silent failure.

**Additional improvement:** After getting the raw path from `GetPath`, attempt to convert it to a canonical path that handles extended-length paths. Use `Path::new(&target)` which Rust's stdlib handles correctly for paths over 260 chars when the system has `LongPathsEnabled`.

### Anti-Patterns to Avoid

- **Calling `canonicalize` on every WalkDir entry:** Expensive. Pre-normalize the excluded list once; use string prefix comparison for entries.
- **Using `rayon::scope` with GDI calls and COM on the same rayon thread:** GDI calls (`extract_icon_png`) and COM calls (`resolve_lnk`) use different thread-local state. Keep them on separate threads (GDI in rayon pool, COM in dedicated thread).
- **Calling `CoUninitialize` without checking the `CoInitializeEx` return value:** If `CoInitializeEx` returns `RPC_E_CHANGED_MODE` (error), calling `CoUninitialize` is incorrect. Only uninit if init returned `S_OK` or `S_FALSE`.
- **Setting `max_depth(1)` for Start Menu:** Start Menu has nested publisher subdirectories. Depth 1 would miss most entries. Use 8.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Bounded thread pool | Manual Vec<JoinHandle> + Mutex<Queue> | rayon::ThreadPool | rayon handles work-stealing, wakeup, join semantics |
| Path normalization | Custom separator-replacement regex | Path::canonicalize + to_lowercase | OS-authoritative; handles junction points, UNC paths |
| WalkDir depth limit | Manual depth counter in recursion | WalkDir::max_depth() | Built into walkdir 2.x; zero extra code |

**Key insight:** The concurrency and path-handling primitives needed for all five items are already provided by existing dependencies (rayon) or stdlib. No custom data structures are needed.

## Common Pitfalls

### Pitfall 1: rayon::scope blocks — plan the call site
**What goes wrong:** `rayon::scope` does not return until all spawned tasks complete. If called from `run_full_index` which is called synchronously in `lib.rs` setup, startup blocks until all icons are extracted. On a fresh install with 200+ apps this could add seconds.
**Why it happens:** `scope` is a join point by design — it guarantees all work completes.
**How to avoid:** Two options: (a) use `pool.spawn()` (fire-and-forget) + store pool as long-lived managed state; or (b) accept blocking in `run_full_index` since it is already called in a spawned thread from `reindex()` (but in `lib.rs` setup it runs on the main thread). Check whether startup blocking is acceptable. Current code already fires threads without joining — matching fire-and-forget semantics requires `pool.spawn()` not `scope`.
**Warning signs:** Startup takes much longer on machines with many apps.

### Pitfall 2: canonicalize fails for non-existent excluded paths
**What goes wrong:** `Path::canonicalize` returns `Err` if the path does not exist. User may have configured an excluded path that has since been deleted. Returning `Err` and using the raw path as fallback is correct but must be explicit.
**Why it happens:** `canonicalize` is a filesystem syscall; non-existent paths error.
**How to avoid:** Use `unwrap_or_else(|_| p.to_path_buf())` as fallback. Still apply `.to_string_lossy().to_lowercase()` to the fallback.
**Warning signs:** Exclusions for deleted directories stop working silently.

### Pitfall 3: COM worker thread and allowlist ownership
**What goes wrong:** The COM worker thread needs the `allowlist` to call `resolve_lnk_inner`. The allowlist comes from `Settings` which is passed to `crawl_dir`. The worker thread has its own lifetime.
**Why it happens:** Rust lifetime rules prevent passing a reference across thread boundaries.
**How to avoid:** Clone the allowlist before sending it to the COM worker thread. It is a `Vec<String>` — cloning is cheap.
**Warning signs:** Compile error: "borrowed value does not live long enough" or "cannot send non-Send type across threads."

### Pitfall 4: IShellLinkW::GetPath MAX_PATH is documented API limit
**What goes wrong:** Increasing the buffer to 32,767 does not guarantee the API returns paths longer than 260 chars — the COM interface itself is documented as MAX_PATH-bounded.
**Why it happens:** Shell Link format has its own path storage limitations predating extended-path support.
**How to avoid:** Treat this as a defensive improvement + future-proofing, not a fix for currently truncated paths. Document the API limitation in a comment. The primary value is preventing silent buffer overrun on any implementation that relaxes the limit.
**Warning signs:** Paths over 260 chars still not resolved — this is expected behavior for the COM interface, not a bug in the implementation.

### Pitfall 5: WalkDir follow_root_links default
**What goes wrong:** Setting `follow_links(false)` also disables `follow_root_links` in some mental models. It does not — `follow_root_links` defaults to `true` and is independent.
**Why it happens:** Confusion between root links (the configured root path itself being a symlink) and traversal links (symlinks encountered while walking).
**How to avoid:** Do not set `follow_root_links(false)` — leave it at default `true`. Only set `follow_links(false)`.
**Warning signs:** Additional paths that are symlinks to directories stop being indexed.

### COM Worker Thread (Item 11 — simplified structure)
```rust
// Source: https://learn.microsoft.com/en-us/windows/win32/api/combaseapi/nf-combaseapi-coinitializeex
use windows::Win32::System::Com::{CoInitializeEx, CoUninitialize, COINIT_APARTMENTTHREADED};
use windows::core::HRESULT;

pub struct LnkQuery {
    pub path: PathBuf,
    pub allowlist: Vec<String>,
    pub reply: std::sync::mpsc::SyncSender<Option<PathBuf>>,
}

pub fn spawn_com_worker() -> std::sync::mpsc::SyncSender<LnkQuery> {
    let (tx, rx) = std::sync::mpsc::sync_channel::<LnkQuery>(128);
    std::thread::Builder::new()
        .name("com-lnk-worker".into())
        .spawn(move || {
            let hr = unsafe { CoInitializeEx(None, COINIT_APARTMENTTHREADED) };
            // S_OK (0) or S_FALSE (1) = success; any negative = failure
            let com_ok = hr.0 >= 0;
            for query in rx {
                let result = resolve_lnk_inner(&query.path, &query.allowlist);
                let _ = query.reply.send(result);
            }
            if com_ok {
                unsafe { CoUninitialize(); }
            }
        })
        .expect("failed to spawn COM worker thread");
    tx
}
```

## Open Questions

1. **Icon pool lifetime: fire-and-forget vs. scope**
   - What we know: `rayon::scope` blocks; `pool.spawn` is fire-and-forget
   - What's unclear: Is it acceptable for startup `run_full_index` to block on all icon extraction, or should it remain non-blocking (current behavior)?
   - Recommendation: Match current fire-and-forget behavior with `pool.spawn` and store the pool as a managed state or long-lived variable (possibly in `start_background_tasks`). This avoids a startup regression.

2. **COM worker thread and re-index interaction**
   - What we know: `run_full_index` is called from multiple contexts (startup, manual reindex, watcher trigger)
   - What's unclear: Should the COM worker be created once globally or once per `run_full_index` call?
   - Recommendation: Create once at startup in `start_background_tasks` (or `lib.rs` setup) and pass as managed state, similar to the timer `tx`. This avoids repeated thread creation and keeps COM init/uninit to one pair per process lifetime.

3. **Path normalization for WalkDir entry paths on Windows**
   - What we know: `canonicalize` resolves symlinks; on Windows, `WalkDir` with `follow_links(false)` yields the symlink's path, not the target's path
   - What's unclear: Should symlink paths themselves be excluded if they fall under an excluded root?
   - Recommendation: Yes — normalize the WalkDir entry path (which is the symlink path) against excluded paths. The normalization handles the comparison correctly.
