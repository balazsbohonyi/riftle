# Phase 09.9: Show Launcher on Default Monitor - Research

**Researched:** 2026-06-14
**Domain:** Tauri v2 window positioning, multi-monitor APIs, Windows cursor detection
**Confidence:** HIGH

## Summary

This phase fixes multi-monitor launcher positioning by changing the launcher from using `current_monitor()` (the monitor the hidden window last occupied) to `primary_monitor()` (the Windows primary/default monitor). A new "Show where cursor is" toggle in Settings allows users to override this and have the launcher appear on whichever monitor contains the mouse cursor. The Settings window is also fixed to always center on the primary monitor, replacing its current one-time-only centering behavior.

**Primary recommendation:** Use Tauri v2's built-in `cursor_position()` + `monitor_from_point()` APIs for cursor-following behavior (no additional crate dependencies needed). Pass `follow_cursor` as a boolean parameter from the Vue frontend to the `show_positioned_launcher` Tauri command — consistent with existing architecture where App.vue already tracks settings like `showPath` and `playSound`.

### Pattern 1: Monitor Selection with Fallback Chain
**What:** Hierarchical monitor selection: cursor monitor (if `follow_cursor`) → primary monitor → current monitor → `win.center()` fallback
**When to use:** In `show_positioned_launcher`, every time the launcher is summoned
**Example:**
```rust
// Source: Tauri v2 docs.rs (confirmed 2026-06-14)
// https://docs.rs/tauri/latest/tauri/window/struct.Window.html

let monitor = if follow_cursor {
    // Try to find the monitor containing the cursor
    win.cursor_position()
        .ok()
        .and_then(|pos| win.monitor_from_point(pos.x, pos.y).ok().flatten())
} else {
    None
};

// Fallback chain: cursor_monitor → primary → current → center()
let monitor = monitor
    .or_else(|| win.primary_monitor().ok().flatten())
    .or_else(|| win.current_monitor().ok().flatten());

if let Some(monitor) = monitor {
    let work_area = monitor.work_area();
    let scale_factor = monitor.scale_factor();
    let physical_width = window_width * scale_factor;
    let physical_anchor_height = anchor_height * scale_factor;
    let x = work_area.position.x
        + ((work_area.size.width as f64 - physical_width) / 2.0).round() as i32;
    let y = work_area.position.y
        + ((work_area.size.height as f64 - physical_anchor_height) / 2.0).round() as i32;
    win.set_position(tauri::PhysicalPosition::new(x, y))
        .map_err(|e| e.to_string())?;
} else {
    win.center().map_err(|e| e.to_string())?;
}
```

### Anti-Patterns to Avoid
- **Reading settings from file in `show_positioned_launcher`**: Adds unnecessary I/O on every launcher summon. Instead, pass `follow_cursor` as a parameter from the frontend (which already has settings in memory).
- **Using `windows` crate for cursor detection**: Unnecessary — Tauri v2 has `cursor_position()` and `monitor_from_point()` that handle DPI and cross-platform concerns correctly.
- **Continuous cursor tracking**: The CONTEXT.md explicitly requires "summon time only." Do not poll cursor position.

**Key insight:** Tauri v2 (2.10+) has mature multi-monitor APIs. `monitor_from_point()` was designed for exactly this use case — no need to reach for platform-specific Windows APIs.

## Common Pitfalls

### Pitfall 1: Physical vs Logical Coordinates Confusion
**What goes wrong:** `cursor_position()` returns `PhysicalPosition<f64>` (physical pixels). `monitor_from_point()` takes `f64` — these are also in physical coordinate space. On high-DPI displays, mixing logical and physical coordinates causes positioning off by the scale factor.
**Why it happens:** Windows `GetCursorPos` returns physical coordinates, but some APIs expect logical. Tauri normalizes everything to physical.
**How to avoid:** Always use `monitor.work_area()` (physical) with `window_width * monitor.scale_factor()` (physical width). The existing `show_positioned_launcher` already does this correctly.
**Warning signs:** Window appears on wrong monitor or offset by ~2x on 200% DPI displays.

### Pitfall 2: `cursor_position()` Unavailable in Non-Desktop Builds
**What goes wrong:** `cursor_position()` is gated behind `#[cfg(desktop)]` in Tauri. Calling it in tests or mobile builds will fail to compile.
**Why it happens:** The method is documented as "Available on desktop only."
**How to avoid:** The `show_positioned_launcher` command runs within the Tauri desktop runtime. Add `#[cfg(desktop)]` guard if extracting cursor logic into a separate function. Unit tests should mock or skip cursor-dependent paths.
**Warning signs:** Compile errors about `cursor_position` not found in non-desktop targets.

### Pitfall 3: Settings Window Size When Hidden
**What goes wrong:** `win.inner_size()` might return `PhysicalSize { width: 0, height: 0 }` for a hidden window, breaking centering math.
**Why it happens:** Some window managers report 0-size for hidden windows.
**How to avoid:** Use the known settings window dimensions as fallback: `win.inner_size().unwrap_or(PhysicalSize::new(450, 600))`. The settings window is configured at 450×600 in `tauri.conf.json`.
**Warning signs:** Settings window appears at `(0,0)` or top-left corner instead of centered.

### Pitfall 4: `SettingsCentered` Removal Impact
**What goes wrong:** Removing `SettingsCentered` managed state registration could shift indices for other managed states accessed via `app.state::<T>()`.
**Why it happens:** Tauri managed state uses type-based lookup (order-independent), not positional. Removing `SettingsCentered` does NOT affect other managed states.
**How to avoid:** Safe to remove — the `setup()` function registers `SettingsCentered` on line 357. Simply delete that line and the struct definition on line 16. All other `.state::<T>()` calls use distinct types.
**Warning signs:** Compile errors about missing `SettingsCentered` type (if references remain) — should not occur since only used in `show_settings_window` and `open_settings_window`.

```rust
// Source: Tauri v2 docs.rs confirmed 2026-06-14
// https://docs.rs/tauri/latest/tauri/window/struct.Window.html#method.cursor_position
// https://docs.rs/tauri/latest/tauri/window/struct.Window.html#method.monitor_from_point

#[cfg(desktop)]
fn resolve_monitor(win: &tauri::WebviewWindow, follow_cursor: bool) -> Option<tauri::Monitor> {
    if follow_cursor {
        if let Ok(cursor) = win.cursor_position() {
            if let Ok(Some(monitor)) = win.monitor_from_point(cursor.x, cursor.y) {
                return Some(monitor);
            }
        }
    }
    // Fall back to primary monitor
    win.primary_monitor().ok().flatten()
}
```

## Open Questions

1. **`cursor_position()` coordinate type precision — logical vs physical**
   - What we know: Tauri docs say it returns `PhysicalPosition<f64>`. On Windows, cursor coordinates from `GetCursorPos` are physical pixels.
   - What's unclear: Whether `monitor_from_point(f64, f64)` expects physical or logical on all platforms. On Windows, physical should work — and this is Windows-only code.
   - Recommendation: Test on a multi-monitor, mixed-DPI setup (e.g., 4K @ 200% + 1080p @ 100%) to verify positioning. Add a note to manual verification checklist.

2. **Settings window `inner_size()` behavior when hidden**
   - What we know: `tauri.conf.json` configures settings window at 450×600 logical pixels.
   - What's unclear: Whether `win.inner_size()` returns valid dimensions when the window is hidden (after `win.hide()` not `win.close()`).
   - Recommendation: Use `win.inner_size().unwrap_or(PhysicalSize::new(450, 600))` as defensive fallback. Manual test: hide settings window, re-open via Ctrl+, — verify it centers correctly.

3. **Monitored state removal order safety**
   - What we know: Tauri managed state is type-keyed, not index-keyed. Removing `SettingsCentered` registration does not affect `SettingsCloseBehavior`, `HotkeyCaptureActive`, or other managed states.
   - What's unclear: Whether any code outside `lib.rs` (e.g., tests, other modules) references `SettingsCentered`.
   - Recommendation: Before removing, grep for `SettingsCentered` across the entire codebase. If only `lib.rs` references exist, safe to remove.
