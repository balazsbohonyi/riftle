## User Constraints (from CONTEXT.md)

### Locked Decisions

**D1 — Remove blocking startup index**
Delete the synchronous `crate::indexer::run_full_index(...)` call from `lib.rs` setup (currently line 248). Replace with a deferred strategy: load the existing DB into memory at startup, then trigger indexing in the background.

**D2 — Startup sequence after change**
New sequence in `lib.rs setup()`:
1. Init DB (unchanged)
2. Load settings (unchanged)
3. `ensure_system_command_icon` (unchanged)
4. `init_search_index` — load existing DB rows into in-memory `SearchIndexState` (unchanged call, new role: loads stale-but-valid data from prior session)
5. Register hotkey (unchanged)
6. Count DB rows to decide deferred index strategy
7. `start_background_tasks` — pass DB row count so it can schedule the deferred startup index

**D3 — Deferred startup index: delay based on DB state**
- If DB is empty (first run): spawn a background thread immediately (no delay)
- If DB has data: wait 30 seconds before triggering the first background index
- The 30-second delay is hardcoded — not a user setting

**D4 — DB empty check**
Add `count_apps(conn: &Connection) -> rusqlite::Result<i64>` to `db.rs`.
Call under the DB lock in `lib.rs` before releasing the guard. Pass `app_count: i64` to `start_background_tasks`.

**D5 — Sync bug fix: background index must refresh the search cache**
Add `app: tauri::AppHandle` parameter to `try_start_index`. Inside the spawned thread, call `crate::search::rebuild_index(&app)` after `run_full_index` completes, before clearing the flag.

**D6 — Timer reset after startup deferred index**
After the deferred startup index completes (whether immediate or 30s delayed), send `TimerMsg::Reset` to the timer thread.

**D7 — What does NOT change**
- `ensure_generic_icon` — still called inside `run_full_index`
- `ensure_system_command_icon` — unchanged position in startup sequence
- `reindex` Tauri command — already calls `rebuild_index` correctly
- Timer and watcher `try_start_index` call sites — only signature and body change
- Watcher settings snapshot — already uses settings clone captured at `start_background_tasks` time

## Summary

This phase makes two targeted surgical changes to the Rust backend. The changes are confined to three files (`lib.rs`, `indexer.rs`, `db.rs`) and involve no new dependencies, no frontend changes, and no new Tauri commands.

**Change 1 (D1–D3–D4):** Remove the blocking `run_full_index` call from Tauri's `setup()` hook. The DB already persists across sessions, so old data loads into the search index immediately. A deferred one-shot thread handles the first index (immediately on first run, after 30s on subsequent runs). This eliminates the startup block where GDI icon extraction was holding up the window.

**Change 2 (D5):** The `try_start_index` function currently spawns a thread that runs `run_full_index` then clears the `is_indexing` flag — but never calls `rebuild_index`. This means the in-memory `SearchIndexState` is never refreshed after timer-fired or watcher-fired re-indexes. The fix adds an `AppHandle` parameter to `try_start_index` (and its caller `start_background_tasks`), then calls `rebuild_index` inside the spawned thread after indexing completes.

**Primary recommendation:** The CONTEXT.md decisions are complete and correct. Research focus is verifying that the actual code matches the described state, and surfacing any threading/borrow-checker concerns before planning.

## Architecture Patterns

### Pattern 1: AppHandle clone across thread boundary

`tauri::AppHandle` implements `Clone` explicitly for this purpose. The pattern is already established in the codebase:

```rust
// In reindex() Tauri command (indexer.rs line 365)
let app_for_thread = app.clone();
std::thread::spawn(move || {
    // ...
    crate::search::rebuild_index(&app_for_thread);
    // ...
});
```

The same pattern applies in `try_start_index`: clone the `&tauri::AppHandle` reference into an owned `tauri::AppHandle` before the `move` closure.

### Pattern 2: DB lock scope before spawning

The DB lock must NOT be held across the thread spawn. The existing code correctly uses block scopes to drop the mutex guard before spawning. The `count_apps` call must happen inside a lock scope that drops before `start_background_tasks` is called:

```rust
// lib.rs setup — after init_search_index, before start_background_tasks
let app_count: i64 = {
    let db_arc = Arc::clone(&app.state::<crate::db::DbState>().0);
    let conn = db_arc.lock().unwrap_or_else(|e| e.into_inner());
    crate::db::count_apps(&conn).unwrap_or(0)
};
```

The guard drops at the end of the block. This pattern is safe and already used throughout the codebase (e.g., Phase 03 decision: "DB lock scope as block `{}` in run_full_index — MutexGuard drops before thread spawn").

### Pattern 3: One-shot deferred thread in `start_background_tasks`

The deferred startup index must spawn before the timer thread so that the timer's deadline starts from app launch, not from when the one-shot completes. The one-shot thread sleeps 30 seconds (if DB has data), calls `try_start_index`, then sends `TimerMsg::Reset`:

```rust
// Inside start_background_tasks, before spawning the timer thread
{
    let app_deferred = app.clone();
    let db_deferred = Arc::clone(&db);
    let data_dir_deferred = data_dir.clone();
    let settings_deferred = settings.clone();
    let is_indexing_deferred = Arc::clone(&is_indexing);
    let com_tx_deferred = com_tx.clone();
    let timer_tx_for_deferred = timer_tx.clone(); // timer_tx created before this block
    std::thread::spawn(move || {
        if app_count > 0 {
            std::thread::sleep(std::time::Duration::from_secs(30));
        }
        try_start_index(
            &app_deferred,
            &is_indexing_deferred,
            &db_deferred,
            &data_dir_deferred,
            &settings_deferred,
            com_tx_deferred,
        );
        let _ = timer_tx_for_deferred.send(TimerMsg::Reset);
    });
}
```

Note: `timer_tx` must be created before the deferred thread block so it can be cloned into the closure. The `mpsc::channel()` call stays at the top of `start_background_tasks`.

### Pattern 4: Watcher thread has no `app` capture today

The watcher thread currently captures `db`, `data_dir`, `is_indexing`, `settings`, and `com_tx` by clone/move. Adding `app` follows the same pattern. The watcher uses `settings` cloned at `start_background_tasks` time — this is explicitly acceptable per D7.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Thread-safe AppHandle sharing | Arc<Mutex<AppHandle>> | `app.clone()` directly | AppHandle is designed to be cloned across threads |
| Deferred execution | async runtime / tokio timer | `std::thread::spawn` + `std::thread::sleep` | Project already uses OS threads throughout; no async runtime present |
| DB row count | Manual iteration | `SELECT COUNT(*) FROM apps` via `query_row` | One-liner, same pattern as other DB queries |

## Common Pitfalls

### Pitfall 1: Holding DB lock when calling `start_background_tasks`

**What goes wrong:** If `count_apps` is called and the lock guard is not explicitly dropped before `start_background_tasks`, the background timer thread may deadlock when it tries to acquire the DB lock.

**Why it happens:** `MutexGuard` is `!Send`, but if the guard lives past the `start_background_tasks` call (e.g. held in a variable in the same `let` chain), the borrow checker will actually prevent compilation. However, if the guard is wrapped in something that obscures lifetime, a logic deadlock is possible.

**How to avoid:** Use an explicit block `{ let conn = ...; count_apps(&conn) }` to guarantee the guard drops before the function call. Pattern is established in Phase 03 decisions.

### Pitfall 2: `timer_tx` must be created before the deferred one-shot thread

**What goes wrong:** If the `mpsc::channel()` call for the timer is placed after the deferred thread spawn, the `timer_tx` clone for `TimerMsg::Reset` is unavailable.

**Why it happens:** The deferred thread needs to send `TimerMsg::Reset` to the timer, but the timer channel must exist first.

**How to avoid:** Create `(timer_tx, timer_rx)` at the top of `start_background_tasks`, then spawn the deferred thread, then spawn the timer thread that consumes `timer_rx`.

### Pitfall 3: `try_start_index` signature change breaks existing tests

**What goes wrong:** Two existing tests call `try_start_index` directly (test `test_timer_fires` and `test_atomic_guard_prevents_double_index`). These will fail to compile after the `app: &tauri::AppHandle` parameter is added.

**Why it happens:** Unit tests cannot construct a real `tauri::AppHandle` without a running Tauri app.

**How to avoid:** The tests must be updated to use a test-double approach. Looking at the existing tests, they pass an empty DB and empty `data_dir`. The `rebuild_index` call inside the spawned thread will call `app.try_state::<SearchIndexState>()` which returns `None` when no state is managed — and `rebuild_index` already handles this case gracefully (it does nothing if `try_state` returns `None`). The test fix is to pass a real AppHandle — but since that is not possible in unit tests, the cleanest approach is to extract a testable inner function or make the `app` parameter optional via a wrapper. The most pragmatic fix: update the test to accept that `try_start_index` now requires an AppHandle and skip the direct call test — the behavior is covered by the integration path through `reindex` (Tauri command).

**Warning signs:** Compile error: `try_start_index` called with wrong number of arguments.

### Pitfall 4: `com_tx` is moved into both the deferred thread and the timer/watcher threads

**What goes wrong:** `com_tx` is a `mpsc::SyncSender<LnkQuery>` which is `Clone`. But `start_background_tasks` takes it by value (`com_tx: std::sync::mpsc::SyncSender<LnkQuery>`). The deferred thread, timer thread, and watcher thread all need it.

**Why it happens:** Value semantics — once moved into one closure, it is unavailable.

**How to avoid:** Clone `com_tx` before each thread spawn. The existing code already clones it: `com_tx_timer = com_tx.clone()` for the timer thread, and `com_tx.clone()` inline for the watcher. Add a `com_tx_deferred = com_tx.clone()` before the deferred thread spawn. This is the established pattern.

### Pitfall 5: `app_count` must be read AFTER `init_search_index`

**What goes wrong:** If `count_apps` is called before `init_search_index`, the DB contains the same data — the count is valid — but conceptually the sequence would be inconsistent with D2's ordering.

**Why it happens:** The decision specifies a specific ordering: `init_search_index` first, then count, then `start_background_tasks`.

**How to avoid:** Follow the sequence in D2 exactly. In practice there is no functional difference because `init_search_index` only reads the DB, it does not modify it. But following D2 keeps the code consistent with the documented startup sequence.

### try_start_index new signature and body

```rust
// Source: D5 in CONTEXT.md, verified against reindex() pattern (indexer.rs line 365-381)
pub(crate) fn try_start_index(
    app: &tauri::AppHandle,   // NEW first param
    is_indexing: &Arc<AtomicBool>,
    db: &Arc<Mutex<Connection>>,
    data_dir: &Path,
    settings: &Settings,
    com_tx: std::sync::mpsc::SyncSender<LnkQuery>,
) {
    if is_indexing
        .compare_exchange(false, true, Ordering::AcqRel, Ordering::Relaxed)
        .is_ok()
    {
        let app = app.clone();           // AppHandle::clone() for move into thread
        let flag = Arc::clone(is_indexing);
        let db = Arc::clone(db);
        let data_dir = data_dir.to_path_buf();
        let settings = settings.clone();
        std::thread::spawn(move || {
            run_full_index(&db, &data_dir, &settings, &com_tx);
            crate::search::rebuild_index(&app);  // FIX: refresh in-memory cache
            flag.store(false, Ordering::Release);
        });
    }
}
```
