# Phase 09.8 — Improve Indexing Performance: Context

## Phase Goal

Remove the blocking `run_full_index()` call from Tauri's `setup()` hook so the app becomes interactive immediately on launch, and fix the sync bug where background indexing updates the database but never refreshes the in-memory search cache.

---

## Decisions

### D1 — Remove blocking startup index

**Decision:** Delete the synchronous `crate::indexer::run_full_index(...)` call from `lib.rs` setup (currently line 248). Replace with a deferred strategy: load the existing DB into memory at startup, then trigger indexing in the background.

**Rationale:** The blocking call sits in Tauri's setup hook, meaning the UI cannot become interactive until the full file-system crawl and GDI icon extraction finish. The DB persists across sessions, so old data is immediately searchable on relaunch.

---

### D2 — Startup sequence after change

New sequence in `lib.rs setup()`:

1. Init DB (unchanged)
2. Load settings (unchanged)
3. `ensure_system_command_icon` (unchanged)
4. `init_search_index` — load existing DB rows into in-memory `SearchIndexState` (unchanged call, new role: loads stale-but-valid data from prior session)
5. Register hotkey (unchanged)
6. Count DB rows to decide deferred index strategy
7. `start_background_tasks` — pass DB row count so it can schedule the deferred startup index

---

### D3 — Deferred startup index: delay based on DB state

**If DB is empty (first run):** Spawn a background thread immediately (no delay). The user would otherwise see no results forever.

**If DB has data:** Wait **30 seconds** before triggering the first background index. This keeps the app snappy at login while still refreshing within a reasonable window.

The 30-second delay is hardcoded — not a user setting.

---

### D4 — DB empty check

Use a count query added to `db.rs`:

```rust
pub fn count_apps(conn: &Connection) -> rusqlite::Result<i64> {
    conn.query_row("SELECT COUNT(*) FROM apps", [], |row| row.get(0))
}
```

Call `count_apps` under the DB lock in `lib.rs` before releasing the guard. Pass the count (`app_count: i64`) to `start_background_tasks`.

---

### D5 — Sync bug fix: background index must refresh the search cache

**Bug:** `try_start_index` spawns a thread that calls `run_full_index` then clears `is_indexing`, but never calls `rebuild_index`. Timer-fired and watcher-fired re-indexes therefore update the SQLite DB but the in-memory `SearchIndexState` is never refreshed — new apps are invisible until manual reindex or restart.

**Fix:** Add `app: tauri::AppHandle` parameter to `try_start_index`. Inside the spawned thread, call `crate::search::rebuild_index(&app)` after `run_full_index` completes, before clearing the flag.

```rust
pub(crate) fn try_start_index(
    app: &tauri::AppHandle,   // NEW
    is_indexing: &Arc<AtomicBool>,
    db: &Arc<Mutex<Connection>>,
    data_dir: &Path,
    settings: &Settings,
    com_tx: std::sync::mpsc::SyncSender<LnkQuery>,
) {
    if is_indexing.compare_exchange(false, true, Ordering::AcqRel, Ordering::Relaxed).is_ok() {
        let app = app.clone();
        let flag = Arc::clone(is_indexing);
        let db = Arc::clone(db);
        let data_dir = data_dir.to_path_buf();
        let settings = settings.clone();
        std::thread::spawn(move || {
            run_full_index(&db, &data_dir, &settings, &com_tx);
            crate::search::rebuild_index(&app);   // FIX: refresh in-memory cache
            flag.store(false, Ordering::Release);
        });
    }
}
```

Propagate `app: &tauri::AppHandle` through `start_background_tasks` (add it as a parameter), which is called from `lib.rs` setup — the `AppHandle` is available there via `app.handle()`.

---

### D6 — Timer reset after startup deferred index

After the deferred startup index completes (whether immediate or 30s delayed), send `TimerMsg::Reset` to the timer thread so the next automatic re-index fires `interval_mins` after the startup index, not from when the app launched.

This is the same pattern already used in the `reindex` Tauri command (manual reindex path).

---

### D7 — What does NOT change

- `ensure_generic_icon` — still called inside `run_full_index`, which is now only called from background threads. No change needed.
- `ensure_system_command_icon` — called once in startup sequence (unchanged position).
- `reindex` Tauri command — already calls `rebuild_index` correctly. No change.
- Timer and watcher `try_start_index` call sites — already exist; only the signature and body of `try_start_index` change (add `app` param, add `rebuild_index` call).
- Watcher settings snapshot — already uses a settings clone captured at `start_background_tasks` time; this is acceptable (same limitation existed before).
