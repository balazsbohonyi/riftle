# Phase 09.1 Research: System Tray + Native OS Context Menu

## Goal Recap
- Show the app in the Windows system tray using the app's default icon.
- Right-click tray icon shows a native OS context menu with the same two items as the launcher context menu: "Settings" and "Quit Launcher".
- Tray should work in dev and release builds.
- Click behaviors and tooltip text are fixed by phase context.

## Non-Negotiable Decisions (from 09.1 context)
- Left-click: toggle launcher (show if hidden, hide if visible).
- Double-click: always show/focus launcher (no toggle on double-click).
- Tray-triggered show must use the same "launcher-show" flow: clear query/results, replay animation, center, focus input.
- If a tray action resolves to show and the launcher is already visible: focus + center, do not hide.
- Tray menu items: exactly two, ordered "Settings" then "Quit Launcher".
- Menu labels must match launcher context menu text exactly.
- Menu actions must match launcher menu behavior (open settings window, quit app).
- Tooltip text: "Riftle Launcher" (static).
- Tray icon uses the app's default/bundled icon.
- Menu must be native OS context menu (no HTML overlay).

## Relevant Existing Code/Assets
- `src/App.vue` provides Settings/Quit actions via `invoke('open_settings_window')` and `invoke('quit_app')`.
- `src-tauri/src/lib.rs` registers `open_settings_window` and `quit_app` commands.
- `src-tauri/src/hotkey.rs` emits `launcher-show` and handles toggle behavior.
- `src-tauri/tauri.conf.json` defines bundle icons (including `icons/icon.ico` and PNGs).
- `src-tauri/capabilities/default.json` lists permissions; no tray permission declared yet.

## Expected Tauri v2 Surface (verify in docs during planning)
- Tray icon creation lives in Rust during `tauri::Builder::setup`.
- Likely module names: `tauri::tray` for tray icon builder/events and `tauri::menu` for native menus.
- There is typically a `TrayIconBuilder` or equivalent that accepts:
  - an `Icon`/resource for the tray image
  - a `Menu` for native context menu
  - tooltip text
  - event handler for click events (left/right/double)
- Tray menu item selection is handled via a menu event callback; actions can call into existing Rust functions/commands.

## How to Source the Default Icon
- Prefer using the app's default window icon loaded from bundle resources.
- If Tauri exposes `app.default_window_icon()` (or similar), use it for the tray icon.
- Otherwise, load the icon resource by path from `tauri.conf.json` bundle icons (most likely `icons/icon.ico` on Windows).
- Goal: no new asset files; reuse existing bundle icon(s).

## Implementation Strategy (Rust)
- Add tray icon creation in `src-tauri/src/lib.rs` inside the desktop `setup` block.
- Build a native `Menu` with two items:
  - `Settings` -> call existing `open_settings_window` logic.
  - `Quit Launcher` -> call `quit_app`.
- Wire tray menu event handler to dispatch the correct action.
- Wire tray click event handler to implement left-click toggle and double-click show rules.

## Show/Hide Flow Integration
- The required "show" behavior is already implemented in `App.vue` under the `launcher-show` event listener.
- Tray code should reuse that by emitting `launcher-show` on show (same as hotkey), even if the window is already visible.
- Hide behavior should call `window.hide()` directly (as hotkey toggle does today).
- For "show" when already visible: call `set_focus` and `center` but do not hide. Emitting `launcher-show` will also re-center and clear input in the Vue handler; this is consistent with "same hotkey-style flow".

## Click Handling Details
- Single left-click: toggle (visible -> hide, hidden -> show).
- Double-click: always show/focus (no hide).
- Avoid double-click triggering single-click toggle as well.
  - If tray API provides `click_count` or a dedicated double-click event, gate logic accordingly.
  - If it emits two single clicks plus a double click, add a small debounce to prevent a hide before the show.
- Right-click should open the native menu automatically (no custom handler needed if menu is attached to tray icon).

## Settings Window Reuse
- `open_settings_window` currently requires `SettingsCentered` state.
- To call it from tray handlers, either:
  - Extract a shared `fn show_settings_window(app: &AppHandle) -> Result<(), String>` used by both the command and tray handler.
  - Or obtain `State<SettingsCentered>` from the `AppHandle` in the tray handler (if API permits).
- Keep the centering-on-first-open behavior intact.

## Permissions / Capabilities Considerations
- Check whether tray functionality in Tauri v2 requires additional capability entries in `src-tauri/capabilities/default.json`.
- If a `tray` permission is required, add it to the default capability for both windows.

## Failure and Edge Cases
- If tray icon creation fails (missing icon, API error), app should still run; log and continue.
- Ensure tray exists in dev builds: do not gate behind `#[cfg(not(debug_assertions))]`.
- Launcher may be hidden at startup; tray icon should still appear.
- If launcher window handle is not found, log and ignore the tray click.

## Validation Architecture
- Manual validation is the primary path (no existing automated UI tests).
- Required checks:
  - Tray icon appears in system tray in both dev and release builds.
  - Tray tooltip reads "Riftle Launcher".
  - Right-click shows native OS menu (not the HTML menu) with exactly two items, ordered "Settings", "Quit Launcher".
  - Clicking "Settings" opens/focuses settings window (single-instance behavior preserved).
  - Clicking "Quit Launcher" cleanly exits the app.
  - Single left-click toggles launcher (show/hide).
  - Double-click always shows and focuses launcher without hiding.
  - Tray-triggered show clears query/results and replays appear animation (same as hotkey).
- Smoke test: verify `launcher-show` event still functions via hotkey after tray integration.

## Planning Notes and Risks
- The exact Tauri v2 tray API and types need to be confirmed during planning; do not assume v1 names.
- If Tauri v2 supports only one click handler, implement explicit debounce or check event metadata.
- Verify how to obtain the default app icon at runtime; prefer APIs over manual file paths.
- Keep tray logic in Rust; no frontend changes should be required for native context menu.
