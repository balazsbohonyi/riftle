# Phase 09.3: Asset Protocol Security Hardening - Research

**Researched:** 2026-03-10
**Domain:** Tauri v2 asset protocol scope, Rust path validation, server-side filename sanitization
**Confidence:** HIGH (all critical findings verified against official Tauri docs, schema, and source code)

---

## Summary

This phase addresses CONCERNS.md High Risk item 5: `assetProtocol.scope` is set to `["**"]` in `tauri.conf.json`, which permits the WebView to request any local file via the `asset://` scheme. If a path-building bug ever exists in the UI or a Tauri command response, an attacker-controlled icon path could read arbitrary files from the local filesystem.

The fix has two independent layers. The first layer constrains the static `tauri.conf.json` scope to the narrowest viable set of glob patterns that still covers both data-directory layouts (installed: `%APPDATA%\riftle-launcher\icons\` and portable: `<exe_dir>\data\icons\`). The second layer adds server-side validation of icon filenames in the Rust search code so that even within the permitted scope, filenames are required to be exactly the hex-hash format generated by `icon_filename()` or one of the two known special names (`generic.png`, `system_command.png`). Together these layers defend in depth.

**The core complication:** `tauri.conf.json` scope patterns are static; they cannot reference a runtime-resolved `data_dir`. The two data-directory roots differ entirely in their prefix, so the scope must cover both. The recommended approach is to use two specific glob patterns — one per mode — rather than one broad pattern. Both patterns constrain access to the `icons/` subdirectory only, so the exposure surface shrinks from "the entire filesystem" to "two specific icons directories".

**Primary recommendation:** Set `assetProtocol.scope` to `["$DATA/riftle-launcher/icons/**", "$EXE/../data/icons/**"]` and add a `validate_icon_filename()` function in `search.rs` with unit tests. The scope patterns cover installed mode (`$DATA` = `{FOLDERID_RoamingAppData}`, i.e., `AppData\Roaming`) and portable mode (`$EXE` = directory containing the exe). No Tauri command changes, no protocol handler changes, no frontend changes required.

## Architecture Patterns

### Recommended Project Structure

No structural changes. Changes are confined to:
```
src-tauri/
├── tauri.conf.json          — scope constraint (2 patterns replace ["**"])
└── src/
    └── search.rs            — add validate_icon_filename() + unit tests
```

No changes to `indexer.rs`, `paths.rs`, `lib.rs`, or any frontend files.

### Pattern 1: Static Scope with Two Directory Glob Patterns

**What:** Two FsScope glob patterns — one for installed mode, one for portable mode — each targeting only the `icons/` subdirectory.

**When to use:** When the data directory has two possible roots that cannot be resolved at config-parse time.

**Tauri FsScope variable mapping (Windows):**
- `$DATA` → `{FOLDERID_RoamingAppData}` = `C:\Users\<user>\AppData\Roaming`
- `$EXE` → path to the application executable (e.g., `C:\path\to\riftle.exe`)

So `$EXE/../data/icons/**` covers the portable layout where `data/icons/` is adjacent to the exe. Note: `$EXE` is the executable file itself, so `..` steps up to the exe's parent directory, then down into `data/icons/`.

**Example (tauri.conf.json):**
```json
"assetProtocol": {
  "enable": true,
  "scope": [
    "$DATA/riftle-launcher/icons/**",
    "$EXE/../data/icons/**"
  ]
}
```

**Confidence:** MEDIUM — `$DATA` resolving to `AppData\Roaming` (not the app-specific subdirectory) is confirmed by the Tauri path API docs (`dataDir()` on Windows = `{FOLDERID_RoamingAppData}`). `$EXE` resolving to the exe path is confirmed by the FsScope variable list in the Tauri config schema. The `../` traversal within a scope pattern is a standard glob behavior, but should be validated with a smoke test before considering the phase complete.

**Risk note:** If `$EXE/../data/icons/**` does not work as expected (some scope implementations canonicalize paths before matching), the fallback is to use `$HOME/*/data/icons/**` or an explicit `$LOCALDATA/../*/data/icons/**` pattern. The smoke test in the verification step will catch this.

### Pattern 2: Server-Side Filename Validation in search.rs

**What:** A pure function `validate_icon_filename(filename: &str) -> bool` that accepts only filenames matching one of:
- Exactly `generic.png` or `system_command.png` (the two static icons)
- Exactly 20 characters: 16 lowercase hex digits + `.png` (the output format of `icon_filename()` in `indexer.rs`)

**Why this format is safe:**
- Rejects any `../`, `/`, `\`, or absolute path prefix — none of these characters appear in the allowed set
- Rejects `.exe`, `.bat`, or any non-`.png` extension
- The 16-hex format is exactly what `FNV-1a → {:016x}` produces — there is no ambiguity

**Where to call it:** In `score_and_rank()` inside `search.rs` when constructing `icon_path` for each `SearchResult`. If `validate_icon_filename()` returns false, substitute `"generic.png"` as a safe fallback.

**Example (search.rs):**
```rust
// Source: this codebase — search.rs score_and_rank()
pub fn validate_icon_filename(filename: &str) -> bool {
    // Allow the two static icons
    if filename == "generic.png" || filename == "system_command.png" {
        return true;
    }
    // Require exactly 16 lowercase hex chars + ".png" = 20 chars total
    if filename.len() != 20 {
        return false;
    }
    let (hex_part, ext) = filename.split_at(16);
    ext == ".png" && hex_part.chars().all(|c| c.is_ascii_hexdigit() && !c.is_uppercase())
}
```

The validation is then applied where `icon_path` is assigned in `score_and_rank()`:
```rust
icon_path: {
    let raw = app.icon_path.clone()
        .unwrap_or_else(|| "generic.png".to_string());
    if validate_icon_filename(&raw) { raw } else { "generic.png".to_string() }
},
```

**Confidence:** HIGH — this is a pure Rust function, no external dependencies, behavior is fully deterministic.

### Anti-Patterns to Avoid

- **Using `$APPDATA` expecting it to resolve to `AppData\Roaming\riftle-launcher\`**: It resolves to `AppData\Roaming\com.riftle.launcher` (bundle-id-based). The actual data dir uses `riftle-launcher` (hardcoded in `paths.rs`), not the bundle identifier.
- **Using a single `$DATA/**` pattern**: This exposes all of `AppData\Roaming`, granting read access to other applications' data.
- **Calling `app.asset_protocol_scope().allow_directory()` at runtime**: The Tauri runtime scope-extension API is intentionally not recommended for security reasons. The static config approach is the correct pattern.
- **Adding validation only in `indexer.rs`**: Validation must be in `search.rs` at result-construction time, because that is the point where filenames from the DB are used to build URLs that reach the frontend. Indexer-side validation would miss DB corruption or direct DB manipulation.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Path component parsing for traversal detection | Manual string scanning for `../` | `std::path::Path::components()` | Handles Windows path separators, normalization edge cases, UNC paths |
| Glob pattern matching | Custom wildcard matcher | Tauri's built-in `assetProtocol.scope` enforcement | Scope enforcement happens in Tauri's asset protocol handler before the response is served — you don't own this layer |

**Key insight:** The validation in Rust does not need to re-implement glob matching. Its job is narrow: verify the filename is exactly the shape produced by `icon_filename()` or is one of two known static names. String length + character-set check is sufficient and has no edge cases.

## Common Pitfalls

### Pitfall 1: $APPDATA Mismatch

**What goes wrong:** Developer uses `"$APPDATA/icons/**"` thinking it matches `AppData\Roaming\riftle-launcher\icons\`. It doesn't. Tauri's `$APPDATA` in FsScope resolves to `AppData\Roaming\com.riftle.launcher\icons\` (bundle-identifier-derived). Icons fail to load silently (403 from asset protocol).

**Why it happens:** The Tauri path API variable `$APPDATA` means "app data dir" (which includes the bundle identifier), not the Windows `%APPDATA%` environment variable. The names are confusingly similar.

**How to avoid:** Use `$DATA/riftle-launcher/icons/**` where `$DATA` resolves to `AppData\Roaming` (the bare Roaming folder), then append the hardcoded `riftle-launcher` subdirectory.

**Warning signs:** Icons load in dev mode but fail in production, or load when running from source but not from installed binary.

### Pitfall 2: $EXE Pattern with Path Normalization

**What goes wrong:** `$EXE/../data/icons/**` might not match if Tauri's scope engine canonicalizes the path before matching (resolving `..` to the actual directory). In that case the pattern resolves correctly but the normalized form might differ.

**Why it happens:** Some glob libraries normalize paths before matching, removing `..` components. If so, `$EXE/../data/icons/**` becomes `<exe_parent_dir>/data/icons/**` after normalization.

**How to avoid:** Run the smoke test: launch in portable mode, confirm an icon loads. If it fails, replace `$EXE/../data/icons/**` with a broader pattern like `$EXE/../../**` (less ideal) or use an explicit absolute path format.

**Warning signs:** Icons load in installed mode but not portable mode.

### Pitfall 3: Validation Applied Only at Write Time, Not Read Time

**What goes wrong:** Adding `validate_icon_filename()` only in `indexer.rs` when writing icon filenames to the DB. This doesn't protect against a DB that has already been written with invalid filenames, or against direct DB modification.

**Why it happens:** Thinking "if the indexer only ever writes valid filenames, the search results will always be valid." This ignores the threat model (path-building bug in the command response layer).

**How to avoid:** Validation must be in `search.rs` at `SearchResult` construction time, as the last line of defense before the filename reaches `convertFileSrc()` in the frontend.

### Pitfall 4: Uppercase Hex Digits Breaking Validation

**What goes wrong:** `icon_filename()` uses `{:016x}` (lowercase hex). If the validation function uses `.is_ascii_hexdigit()` without checking for uppercase exclusion, it would incorrectly accept `"ABC1234567890123.png"` which would never appear from normal indexing, creating a mismatch between what's allowed and what's generated.

**Why it happens:** `is_ascii_hexdigit()` in Rust returns true for both upper and lowercase hex characters.

**How to avoid:** Check `!c.is_uppercase()` in addition to `is_ascii_hexdigit()`, or use a more specific check: `matches!(c, '0'..='9' | 'a'..='f')`.

```rust
#[cfg(test)]
mod tests {

    #[test]
    fn test_validate_icon_filename_accepts_valid_hex() {
        // Valid: exactly 16 lowercase hex + ".png"
        assert!(validate_icon_filename("abc1234567890def.png"));
        assert!(validate_icon_filename("0000000000000000.png"));
        assert!(validate_icon_filename("ffffffffffffffff.png"));

        assert!(validate_icon_filename("system_command.png"));
    }

    #[test]
    fn test_validate_icon_filename_rejects_traversal() {
        assert!(!validate_icon_filename("../secret.png"));
        assert!(!validate_icon_filename("../../etc/passwd"));
        assert!(!validate_icon_filename("..\\system32\\evil.exe"));
    }

    #[test]
    fn test_validate_icon_filename_rejects_absolute_paths() {
        assert!(!validate_icon_filename("/etc/passwd"));
        assert!(!validate_icon_filename("C:\\Windows\\System32\\cmd.exe"));
    }

    #[test]
    fn test_validate_icon_filename_rejects_wrong_extension() {
        assert!(!validate_icon_filename("abc1234567890defg.exe"));
        assert!(!validate_icon_filename("abc1234567890defg.jpg"));
        assert!(!validate_icon_filename("abc1234567890defg"));
    }

    #[test]
    fn test_validate_icon_filename_rejects_uppercase_hex() {
        // icon_filename() always produces lowercase; uppercase would be anomalous
        assert!(!validate_icon_filename("ABC1234567890DEF0.png"));
    }

    #[test]
    fn test_validate_icon_filename_rejects_wrong_length() {
        assert!(!validate_icon_filename("abc.png"));
        assert!(!validate_icon_filename("abc1234567890defghij.png")); // 20 hex + ".png" = 24 chars
    }
}
```

## Open Questions

1. **Does `$EXE/../data/icons/**` resolve correctly in Tauri's scope engine?**
   - What we know: `$EXE` is documented as a valid FsScope variable. `..` in glob patterns works in standard glob matching.
   - What's unclear: Whether Tauri normalizes the path before checking the scope, and whether normalization would change matching behavior.
   - Recommendation: Add a manual smoke test as part of APSH-01 verification: launch in portable mode (with `riftle-launcher.portable` marker), search for any app, confirm the icon renders. If it fails, replace the pattern with `$HOME/**/data/icons/**` as a fallback.

2. **Does the CSP `img-src` restriction need updating?**
   - What we know: The current CSP is `img-src 'self' asset: http://asset.localhost data: blob:`. This permits any `asset://` URL. The scope change in `assetProtocol` constrains which filesystem paths can be requested via `asset://`, but the CSP itself doesn't need to change — `asset:` is still the right scheme for `convertFileSrc()`.
   - What's unclear: Nothing — the CSP and the scope are two independent mechanisms and both are already correct for the intended behavior.
   - Recommendation: No CSP change required.
