# Phase 09.3 Plan 03: portable-safe validated icon loading

**Portable and installed launcher icons now load through a validated Rust command with frontend caching, eliminating the brittle Windows `$EXE` asset-scope dependency without reopening wildcard filesystem access.**

## Accomplishments
- Added `get_icon_bytes` plus helper coverage so icon file reads stay under `{data_dir}/icons`, reject invalid filenames, and fall back to `generic.png` for missing app icons
- Replaced `convertFileSrc()` icon loading in `App.vue` with cached blob URLs backed by the Rust command so repeated search results do not refetch the same bytes
- Removed the obsolete `assetProtocol` scope and `protocol-asset` feature because app-managed icons no longer require direct asset:// access

## Files Modified
- `src-tauri/src/commands.rs` - Added validated icon-byte loading helpers, Tauri command, and unit tests
- `src-tauri/src/lib.rs` - Registered `get_icon_bytes` in the invoke handler
- `src/App.vue` - Switched icon rendering to async cached Rust-backed blob URLs
- `src-tauri/tauri.conf.json` - Removed asset protocol scope and kept CSP to `data:`/`blob:` images only
- `src-tauri/Cargo.toml` - Dropped the `protocol-asset` Tauri feature
- `src-tauri/Cargo.lock` - Refreshed lockfile after dependency feature cleanup

## Decisions Made
- Use raw PNG bytes returned from Rust and converted to blob URLs in the renderer instead of widening filesystem scope again
- Keep missing-icon fallback inside Rust so the frontend always has one retrieval path for both app icons and `generic.png`

## Deviations from Plan
- Replaced the planned “fix the portable asset scope” direction with a stricter Rust-served loading path after confirming Tauri’s Windows `$EXE` resolution is not reliable for this use case
